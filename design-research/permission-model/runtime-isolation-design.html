Runtime 物理隔离 — daemon vs sandbox vs cloud microVM

锚定 核心不变量（bible） · 这是「物理层」防御，bible 是「逻辑层」

这份文档讲什么

Bible 的 permission / audit / accountability 是逻辑层防御。这份是物理层防御 — agent 进程实际跑在哪里？跟 user 系统其他进程怎么隔离？跑出失控 tool call 时，OS 能挡住吗？

核心张力："daemon-on-user-machine" 哲学（agent 跑在 user 自己机器，便宜、白盒、低延迟）vs "strong isolation"（agent 跑在隔离 VM，安全、可承受不可信代码）。这两者本质冲突，没有银弹。本文给两条 path + 一个 hybrid 推荐。

物理隔离不是孤立维度：「agent 跑哪台机器」直接被 bible 的 INV-2（workspace hard wall） 和 INV-4（machine 的 owner + collaborator） 约束 —— 见 §3 ISO-4。

1. 现状摘要 — 隔离强度参差不齐

1.1 底座是可扩展集合，sandbox 行为参差不齐

Alice 当前支持 11 个底座 CLI：claude、codex、opencode、openclaw、hermes、gemini、pi、cursor、copilot、kimi、kiro。底座注册是 declarative — 加新底座只是加一行 probe()。下面是每个底座 × OS 的实际隔离强度：

底座（按 sandbox 成熟度分组）Linux 实际隔离macOS 实际隔离
Codex（唯一有自带 sandbox 的） ✅ Landlock workspace-write + network_access (硬约束) ❌ fallback 到 danger-full-access — Seatbelt bug (openai/codex#10390)
Claude（自带 per-tool approval，无 OS sandbox） ❌ 无 OS sandbox ❌ 无 OS sandbox
其他 9 个（opencode / openclaw / hermes / gemini / pi / cursor / copilot / kimi / kiro） ❌ 无 OS sandbox ❌ 无 OS sandbox
扩展性原则 底座的 isolation 设计要做统一抽象 — 不绑死特定列表。未来加任何新底座（注册一行 probe）都自动继承 sandbox / per-task home / memory pointer 这套机制。不应该为每个底座写 hardcoded isolation 逻辑。
⚠ 最严重的事实 公司里一半员工用 Mac，macOS 上跑 codex 就是裸跑。`server/internal/daemon/execenv/codex_sandbox.go:60-80` 明文写："codex on macOS: seatbelt ignores sandbox_workspace_write.network_access → fall back to danger-full-access so the Alice CLI can reach the API"。Linux 上的 Landlock 是真硬约束；Mac 上的"沙箱"实际是没沙箱。

  • DONE Per-task CODEX_HOME：避免 codex 看到系统级 ~/.codex/skills/

  • MISSING macOS 真 sandbox
  • MISSING Cloud runtime 路径

2. Assumptions

  1. A1 — Agent runtime ≠ trusted code。Agent 拿到 LLM 输出后会真的执行 tool call。即使 LLM 是合作的，prompt injection / 错误推理可能导致 destructive command。Runtime 必须假设里面跑的代码不可信。
  2. A2 — User 机器是已经被信任的环境。User 的 mac 上已经有他的 git config、SSH key、cloud token、浏览器 cookie。agent 跑在这台机器上，跟 user 的其他进程是 peer，不是 hostile 邻居。
  3. A3 — 公司内部信任（同主设计 A1）。同事写的 agent 不是 hostile，但会 fail-open（agent 不是恶意的，但可能犯错）。
  4. A4 — Agent 行为 emergent。Capability 预先列不全，靠 sandbox 兜底（防御 agent 没预想到的 destructive action）。
  5. A5 — Latency 敏感。Agent task 启动 cold start > 5s 用户感觉糟糕；> 30s 不可用。这是大多数 cloud microVM 方案的死穴。
  6. A6 — Cost 敏感。Alice 内部工具不应该比 user 自己跑 codex CLI 贵 10 倍。Cloud sandbox 每分钟 0.01 美元，密集使用一天就 $10+/agent。
  7. A7 — Mac > Linux > Windows 是用户实际分布。公司绝大多数员工用 Mac，少数 Linux dev，几乎没人 Windows。设计要照顾 Mac。

3. Invariants — 物理隔离的三条原则

ISO-1Filesystem 边界是底线

陈述：无论选哪种 sandbox 方案，agent 进程能写的文件路径必须可枚举、可声明。默认只能写 task workdir，其他路径（~/.ssh、~/.aws、~/.alice、prod credentials、user 其他项目）必须显式 grant。

这跟 bible INV-7（Permission ≠ ACL，intrusive filter） 互补而非重复 — INV-7 是 logical 层（"agent 想读 X，我们 detect 并 prompt"），ISO-1 是 physical 层（"agent 进程根本访问不到 X 那个 syscall"）。逻辑层会被 prompt injection 绕过，物理层不会。

ISO-2Network 边界至少 deny-by-default

陈述：Agent 默认只能 outbound 到几个白名单：Alice API、model provider、显式声明的 MCP server。Random outbound（C&C、data exfiltration、bitcoin mining）必须 fail。这直接服务 bible INV-1 的 P1（数字资产） 防护。

当前现状：codex Linux Landlock 支持 network_access = true （allow all）/ false （allow none），没有 allowlist 模式。Mac fallback 后 network 完全开放。这个 invariant 当前 不满足。

ISO-3白盒 > 黑盒，能观测 > 物理隔离强度

陈述：当物理隔离强度 vs 可观测性冲突时，选可观测性。一个完美隔离但开发者看不见在做什么的 sandbox（如 e2b 全 cloud），比一个轻量但 daemon 能看到所有 syscall / file access / network 的 sandbox 更糟。

这跟 bible INV-10 (audit-over-ACL) 一致 — Alice 优先做透明度，不做不可见的强 enforce。Agent 跑在哪、做了什么，都进同一条 audit pipeline。

ISO-4Runtime placement 受 collaboration model 约束，不是纯 ops 决策

陈述：「agent 跑哪台 machine」不能只看延迟/成本，必须先满足 bible 的 INV-2（workspace hard wall） 和 INV-4（machine 的 owner + collaborator）：

  • 同 workspace 才行（hard wall）：agent 只能跑在与它同 workspace 的 machine 上。跨 workspace 借机器 = 破墙，绝对禁止。
  • machine 的 owner/collaborator 决定谁能借它跑：
    • private machine（visibility=private）：只有 machine owner 自己的 agent 能跑。
    • shared machine（visibility=public，owner 显式开放）：该 machine 的 collaborator 的 agent 也能跑。
  • caller 视角的下钻：user U 调用 agent A 跑在 machine M 上，要求 U 至少是 M 的 owner 或 collaborator —— 否则等于「拿别人的机器 + 别人的本地 credentials 跑活」，既破 INV-4 也制造 confused deputy。

含义：Path A（跑 user 机器）天然满足 —— agent owner 调自己的 agent 跑自己的 machine。一旦要「借」别人的机器，placement 决策就必须查 machine 的 owner/collaborator 关系。这也是 Path B（cloud）的一个动机：cloud sandbox 不属于任何员工的 machine，绕开了「借机器」的信任问题。

6. 两条 path + hybrid 推荐

Path A — 守 user-hardware-trust，加强软隔离

"agent 跑用户机器" 这个哲学不变。在 user 机器上做更强的 OS-level sandbox。

具体动作

  • 所有非 codex 底座加 per-task home 隔离（统一抽象），跟 codex CODEX_HOME 同思路（symlink 用户级 skills / config 进 per-task home + 拒绝系统目录）
  • Linux：除 codex 之外的 10 个底座统一用 bubblewrap 包一层 — 跟 codex Landlock 同强度但跨底座统一
  • macOS：写 sandbox-exec sbpl profile，apply 到所有底座启动命令。接受 deprecated 风险作 best-effort
  • Network：用 nftables (Linux) 或 LSP filter (Mac) 把 outbound 限到 allowlist。这层 daemon 启动时 setup，task 进程继承

优劣

优势 ✅ 0 cost ✅ 0 latency ✅ 跟现有 daemon 架构兼容 ✅ user 跑自己机器没心理 friction
劣势 ❌ Mac sandbox-exec deprecated，未来可能彻底没用 ❌ 隔离上限不如 microVM ❌ network allowlist 实现复杂 ❌ user 机器配置漂移导致 sandbox profile 失效

Path B — Cloud microVM，sensitive task opt-in

日常 agent 仍在 user 机器跑（Path A 的 baseline），但 service agent / 大批量数据访问 / destructive command 走 cloud microVM。

具体动作

  • Cloud runtime 服务：alice server 接 e2b 或自建 Firecracker pool
  • Task routing：daemon 根据 agent.permission_mode 和 intrusive filter 决定本地跑还是 cloud
  • 状态同步：cloud sandbox 里 agent 的 workdir / memory pointer 通过 alice sync engine pull/push
  • 触发条件 (默认)：
    • 所有 service mode agent → cloud
    • 普通 agent 任务里包含 destructive tool call → upgrade 到 cloud（owner approve 后）
    • 访问 sensitive tag 资源 → cloud

优劣

优势 ✅ 隔离上限是 industry-grade microVM ✅ 平台独立（user 在 Mac / Linux / Windows 都跑得动） ✅ user 机器宕机也能 continue ✅ 多个 agent 真并行
劣势 ❌ Cold start 1-5s ❌ cost 持续累积 ❌ user 看不到 cloud 进程在做什么（破 ISO-3 白盒原则） ❌ 跟 user 本地状态（credentials / workspace 实际 git checkout）同步复杂 ❌ 离开 daemon-on-user-machine 哲学

不破坏白盒的关键设计

Cloud runtime 不能是黑盒 为满足 ISO-3，cloud sandbox 必须让 user / owner 实时看见里面在做什么。具体：每个 tool call 通过 sync engine push 回 alice，UI 跟 agent 在本地跑时同形态。"agent 跑哪里"是 ops 决策，"agent 在做什么"必须永远 visible。这是 e2b vs Devin 的核心区别 — e2b 提供 sandbox 但 user 控制 UX；Devin 把整个 box 黑了。

Path C (推荐) — Hybrid: Path A 是 default + Path B 是 opt-in

最实际的方案 — 不是二选一。99% 任务跑 Path A（low latency / cost），少数高风险任务走 Path B（强隔离）。

怎么做

# Alice 写一份 sbpl profile (Scheme-like syntax)
(version 1)
(deny default)
(allow process-fork process-exec)
(allow file-read* (subpath "/usr") (subpath "/System") (subpath "/Library"))
(allow file-read* file-write* (subpath "/Users/<user>/alice_workspaces/<task>"))
(allow network-outbound (remote tcp "alice-api.machine-pulse.net:443"))
;; Deny by default for everything else

# Daemon 启动 codex/claude 时
sandbox-exec -f /tmp/alice-task-<id>.sb codex exec ...

  • Deprecated 没新文档，sbpl 语法靠 reverse engineering 已有 profile（如 Apple Quarantine / iOS WebKit）
  • Codex 那个 network_access bug 同样会影响我们 — 但因为我们写自己的 profile，可以避开（明确 allow network-outbound to specific host）
  • 未来 macOS major version 可能彻底废除 — 但短期 (1-3 年) 仍可用

8. 工程设计

8.1 Daemon 启动 agent 的统一抽象

// 不管底座是 11 个里的哪一个（或未来新增的），daemon 用同一个 sandbox 包装
type RuntimeSandbox interface {
    Wrap(cmd *exec.Cmd, policy SandboxPolicy) error
}

type SandboxPolicy struct {
    // ISO-1: filesystem
    WritablePaths []string        // 默认: task workdir
    ReadablePaths []string        // 默认: workdir + 系统 dylib / lib
    DeniedPaths   []string        // 默认: ~/.ssh, ~/.aws, ~/.alice, prod creds

    // ISO-2: network
    NetworkMode   string          // "deny" | "allowlist" | "all"
    AllowedHosts  []string        // alice-api, model provider, declared MCP servers

    // ISO-3: observability — 进 audit pipeline (Graylog)，非应用观测
    AuditSink     io.Writer       // syscall / file / network 事件 → audit event
}

// 平台分发
func NewSandbox(os string) RuntimeSandbox {
    switch os {
    case "linux":  return &BubblewrapSandbox{}
    case "darwin": return &SandboxExecSandbox{}
    default:       return &NoOpSandbox{}  // fallback (with warning)
    }
}

8.2 所有底座如何被包

底座类别当前启动方式加 sandbox 后
Codex（自带 sandbox） codex exec ... 用 codex 自带 Seatbelt/Landlock 外层加一层 sandbox 兜底 — codex 自己的内层 sandbox 仍 active 作为冗余
其他所有底座（claude / opencode / openclaw / hermes / gemini / pi / cursor / copilot / kimi / kiro / 未来新增） 各自 CLI 启动，无 OS sandbox 外层 sandbox = 唯一防御。Linux bwrap / Mac sandbox-exec 启动，所有底座走同一抽象

8.3 Network allowlist 的实现

Linux: nftables / firejail

  • Daemon 启动时 setup nftables rule for task UID
  • Outbound 限 alice-api.machine-pulse.net + api.anthropic.com + 声明的 MCP
  • Task 进程继承 UID，自动应用 rule
  • Task 结束清理 rule

macOS: sbpl network rule

  • sandbox-exec profile 里写 (allow network-outbound (remote tcp "host:port"))
  • 但 sbpl 不支持 hostname allowlist (只接 IP / port)
  • workaround: daemon 启动前 DNS resolve allowlist host，写入 sbpl 当时 IP — 重启时 refresh
  • 不完美但比裸跑好

8.5 UI: 让 user 知道 agent 跑哪里

UX 关键 ISO-3 白盒原则要求 user / owner 永远知道 agent 在哪。Agent profile / issue activity stream 上要有明确标识：
  • 🖥 Local (Path A) — 跑在 user 自己 machine，绿色 indicator
  • ☁ Cloud sandbox (Path B) — 跑在 e2b microVM，黄色 indicator + 提示原因 ("This service agent runs in cloud sandbox")
  • ⚠ Cloud (升级触发) — 因为触发 critical filter 临时升级，红色 indicator + owner 通知

10. 已知风险 + 反例

风险 1 — sandbox-exec 被 Apple 彻底废 Mac 上唯一的 native 选项。Apple 任何 macOS major release 可能拿掉。 缓解：监控 macOS dev beta，把 sbpl profile 当 "best effort"，service agent 强制 cloud，普通 agent 接受 fallback (跟现在 codex fallback 同 risk profile)。
风险 2 — bubblewrap 在 Linux 老 kernel 不可用 某些 LTS distro kernel < 4.4 没 user namespaces。 缓解：daemon 启动时 probe，不可用就 warning + fallback no-op。Alice 文档提醒最低 kernel 版本。
风险 3 — e2b 价格突变 SaaS 单价说涨就涨 (跟 Anthropic API 同 dependency)。 缓解：cloud runtime 抽象 interface，e2b 是其中一个实现，未来切自建 Firecracker / Modal / fly.io 都可以。Sandbox interface 别绑死 e2b API。
风险 4 — network allowlist 维护爆炸 每个 MCP / API / tool 都要进 allowlist。如果 agent 接 50 个 MCP，allowlist 维护成本爆炸。 缓解：allowlist 不是 task-specific 是 agent-specific (在 agent 配置里声明)，agent iteration 时一起 review。所有 declared MCP 自动 add；其他 outbound 强制 prompt owner。
风险 5 — Sandbox 跟 agent 协作工具冲突 sandbox 把 ~/.aws 屏蔽，但 agent 想用 user 的 aws-cli 跑某个 task — agent 直接 fail。 缓解：声明式机制。Agent 在 skill 里可声明 requires: [aws-creds]，daemon 知道这个 task 需要打开 ~/.aws read，prompt user 一次性 grant，写进 task scope。
