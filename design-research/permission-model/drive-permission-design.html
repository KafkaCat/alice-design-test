1. 现状摘要

1.1 已实现 ✅

  • Schema 完整 — drive_space / drive_space_member / drive_folder / drive_file / drive_reference / project_drive_space 六张表
  • Object storage — 后端 multipart proxy 上传，CloudFront signed URL 30 分钟下载
  • Object key 含 workspace_id 前缀 — drive/<workspaceID>/<uuidv7><ext>
  • Nested folder — parent_id 自引用 + lazy load UI
  • Attachment ↔ drive 双向桥 — issue 附件可 save-to-drive，反之亦可
  • drive_reference — file 关联到 issue/comment/chat_message/workflow_run
  • Polymorphic subject — drive_space_member.subject_type ∈ (member, agent)

1.2 设计了但没跑起来 ⚠️

Schema 字段设计意图现状

1.3 关键空白 ❌

最痛的两个空白
  1. 没有任何 member-management endpoint — Schema 把 ACL 设计得很美，运行时一行 add member 的代码都没有
  2. drive_reference 跨 space 时只能黑盒 — 文件挂到 issue 上，issue 看得见的人不一定有 file 权限，当前返回 {restricted: true} 占位，"看得见有附件但拿不到 URL"

2. 设计原点 — Drive 怎么融入主设计

下面 6 条 invariant 是主设计模型在 drive 这个领域的延伸。每条都对应主设计的某条 invariant，但具体到 drive 这个 domain。

DRV-0Sensitive data 通过 workspace 隔离，不靠 file-level ACL

陈述：HR / 财务 / 法务 / 客户合同 / 未发布产品这类敏感数据，当前阶段的主防线是把它放到独立 workspace。Workspace 是 hard wall — agent / user 跨 workspace 默认完全不可达。Workspace 内部走轻量 ReBAC，default 倾向 transparency。

含义：

  • 需要保密的 drive 内容 → 创建独立 workspace（如 hr-workspace、finance-workspace），只对相关员工 grant member
  • 同 workspace 内部不做 file-level ACL，只做 space-level + team-level 区分
  • Agent 跟 user 一样受 workspace hard wall 约束，没有特例

DRV-1Drive 是 P1 保护对象（公司数字资产）

陈述：网盘里的文件 99% 是"公司数字资产"（合同 / 设计稿 / 代码备份 / 客户数据 / 财务）。按 INV-1 优先级，drive 是 P1 第一优先保护对象，跟 git push / DROP TABLE 同级。

含义：

  • 所有 drive read/write 走全量 audit（INV-8，发到独立 audit 工具）
  • Intrusive filter 在 drive read 维度有特殊 rule：批量下载 / 跨 space 访问 / agent 读未声明 path 都属于 intrusive
  • Drive 的 default ACL 比 issue/chat 更紧 — 默认 private space，需要显式 share 到 team/workspace

DRV-2Space-level ACL，无 file-level ACL（INV-7 的领域延伸）

陈述：权限挂在 space 层，不挂在 file 层。这跟 INV-7 "不做 fine-grained capability" 是同一思路在 drive 上的应用。

为什么不做 file-level ACL File 多 = 几千几万。Per-file ACL = 维护噩梦 + UI 灾难 + 心智负担巨大。Space 这一层粒度刚好：一个 space 是一组逻辑相关文件（项目 / team / 私人），ACL 按 space 设置即可。需要更细粒度时用多个 space而不是 file-level ACL（"敏感版" vs "脱敏版" 各开一个 space）。

反对意见 + 我们的回应：

反对回应
"那 share 单个文件给一个人怎么办？" 开一个 private space，加这个人当 viewer，把文件移过去。或者用 drive_reference 挂到 issue 上（继承 issue 权限，DRV-3）
"那 'public link' 分享呢？" 不实现。内部工具不需要 anonymous public link，所有人都是 workspace member

DRV-3Drive 权限走 ReBAC 关系图（含 team 维度）

陈述：用户能否访问一个 file，不只看 drive_space_member 直接 ACL，要走完整的关系图。这跟主设计的轻量 ReBAC 一致 — drive 接入同一关系网，包括 team 关系（team 是同 workspace 内细分可见性的机制）。

访问路径有 4 种

user ─[member of]→ workspace ─[contains]→ drive_space(kind=workspace) ─[contains]→ file ① 全员池 user ─[member of]→ team ─[owns]→ drive_space(kind=team) ─[contains]→ file ② 团队空间 user ─[creator/member of]→ drive_space(kind=private) ─[contains]→ file ③ 私人空间 user ─[can_view]→ issue ←[references]── file (in any space) ④ 跨 space 借用

路径 ④ 是关键 — drive_reference 把 file 挂到 issue 上时，能看 issue 的人就能下载该 file（即使 file 在他没权限的 space 里）。当前代码遇到这种情况返回 {restricted: true} 黑盒 — 这是 bug，要修。

DRV-4Agent 是 first-class drive subject（INV-3 的延伸）

陈述：Agent 跟 user 一样可以是 drive_space_member、可以是 file 的 created_by、可以拥有自己的 agent-output space。这跟 INV-3 "agent 是 first-class citizen" 一致。

具体规则：

  • Agent 默认能访问的 = owner 能访问的 ∩ 当前 caller 能访问的（user_equivalent mode）
  • Agent 跟 user 一样受 team 边界约束 — caller 不在 team T，调用的 agent 也不能访问 team T 内的 file
  • Service mode agent 可以有额外 space membership（admin 声明），但仍受 workspace hard wall 约束
  • Agent 上传文件时 source_type='agent_output'，default folder 是 agent 自己的输出空间
  • Agent 产出的文件 created_by_type='agent', created_by_id=agent.id，但 audit log 记录 owner

DRV-5所有 drive access 全量 audit，发到独立 audit 工具

陈述：每次 file read / write / delete / share 都写 audit event。包括 signed URL 的生成（拿了 URL 就视为 access，无论是否真下载）。

跟 INV-8 一致 — audit-over-ACL，发到独立 audit 工具（Graylog / OpenSearch 类），不写业务 DB。Drive 是 P1 资产 → audit fail-close（写 audit 失败 = 拒绝 access）。

3. 四种 Space 形态 — kind 字段终于有用

Schema 已经有 kind 字段但行为没分化。重新定义四种形态，每种有清晰的 access pattern。

🏢kind = workspace — 全员池

workspace 所有 member 默认 viewer，无需 add to space_member。适合 onboarding / 公司公告 / 公开模板。
  • 谁可建：service_agent_admin 及以上
  • 谁可读：所有 workspace member 自动 viewer
  • 谁可写：space_member 里显式 editor/admin
  • 典型例子：公司 wiki 附件 / 入职文档 / 节日福利照片

👥kind = team — 团队空间

绑定到一个 team，team_member 自动获得 access，role 跟随 team 角色。新增设计，当前 schema 不支持。
  • 谁可建：team owner
  • 谁可读：team_member 自动 viewer
  • 谁可写：team_member.role='owner' 自动 editor，其他需 invite
  • 典型例子：Engineering team 的设计稿 / Sales team 的合同模板
  • 新增字段：drive_space.team_id UUID REFERENCES team(id) + kind='team' check

🔒kind = private — 私人空间

创建者独占。不能 invite 别人。要 share = 把文件移到 team/workspace space 或者通过 issue reference 借用。
  • 谁可建：任何 user
  • 谁可读/写：仅 creator
  • 典型例子：员工自己的草稿 / 个人笔记 / 暂存
  • 员工离职：private space 默认归档（不删除，配合 INV-1 priority 3 员工个人隐私）；HR 触发后管理员可申请解锁但需 user 同意 OR 法律授权

🤖kind = agent_output — Agent 产出池（新增 kind）

每个 agent 自动有一个 agent_output space，agent 在 task 产出的文件默认落这里。owner 是 space admin。
  • 谁可建：agent 创建时 daemon 自动建
  • 谁可读：owner + 该 agent 当前 task 的 caller
  • 谁可写：只有这个 agent 自己（subject_type='agent', subject_id=agent.id, role='editor'）
  • 典型例子：deploy_bot 生成的 deploy 报告 / pr_reviewer 的 review summary / 数据分析 agent 跑的 query 结果
  • 清理策略：90 天无 reference 自动归档；agent archive 时 space 也 archive

📁kind = project — 项目空间（恢复死代码）

project_drive_space 表当前是死代码。可以复活作为 project-bound space，让 project 成员自动获得 access。但优先级低，team space 先做。
  • 谁可建：project lead
  • 谁可读：project 成员
  • 典型例子：某个具体 launch 的所有素材
  • LATER 不是 MVP

4. Agent × Drive 的具体交互

📥Agent 读 file

  • Agent 通过 X-Agent-ID + X-Task-ID 走 resolveActor，identity 是 agent
  • 权限 check：agent.permission_mode='user_equivalent' → 检查 (a) caller user 能读吗 (b) agent 自己作为 subject 能读吗 — 两个都通过才放行
  • service mode → 检查 service 声明范围里是否包含此 space
  • Audit 写：agent_id, owner, caller, file_id, space_id

📤Agent 写 file

  • 默认目标：agent 自己的 agent_output space
  • 显式目标：owner 允许的 space + agent 是 space_member editor 才能写
  • source_type='agent_output', source_id=task.id
  • 大文件（>100MB）触发 intrusive filter prompt owner

🔗Agent 创建 drive_reference

  • Agent 工作时把产出 reference 到 issue（"我把 deploy log 挂到这个 issue 上了"）
  • 跟 sync engine 集成：reference 创建 → 自动推送给所有能看 issue 的人
  • 权限继承走 DRV-3 路径 ④ — issue 可见者自动可下载

⚠Agent 大批量 access

  • "summarize all files in /finance/" — intrusive filter 触发
  • Layer 1 rule: 单 task > 50 file = halt
  • Layer 2 LLM judge: 看 agent 自然语言意图是否合理
  • owner approve 后放行 + flag this agent baseline 允许 bulk

5. drive_reference 跨 space 继承 — 修复"restricted 黑盒"

当前最大 UX 痛点：drive 文件挂到 issue 上，issue 看得见的人不一定能下载文件（因为 file 在他没权限的 space）。当前代码返回 {restricted: true} 黑盒占位 — issue 上看到"有附件"但点不开。

5.1 新规则

drive_reference 是显式权限授予 当 user A 把 file F (在 private space S 里) reference 到 issue I 上时，所有能看 issue I 的人，自动获得 file F 的 viewer 权限。这是 A 的显式授权行为 — 既然他把它挂到 issue 上，就是有意让所有 issue 参与者看到。

5.2 权限 check 逻辑（新）

CanReadFile(user, file):
    # Path 1: 直接 space 权限
    if CanReadSpace(user, file.space) return true

    # Path 2: 通过 drive_reference 借用
    for ref in DriveReferencesOf(file):
        if ref.target_type == 'issue' and CanViewIssue(user, ref.target_id):
            return true
        if ref.target_type == 'comment' and CanViewComment(user, ref.target_id):
            return true
        # 其他 target_type 同理

    return false

5.3 UI 上的透明度

用户看到一个通过 reference 借用的文件时，UI 明确标识：

📎 design-spec.png [download] ⤴ borrowed via issue #1234 "Login redesign" stored in: 🔒 Alice's private space

这样：

  • 权限来源透明（不是黑盒）
  • 原 space owner 看见自己的文件被多少 issue 借用（DRV-5 audit）
  • "我不想再 share 这个文件" → 取消 drive_reference 即可

5.4 撤销 reference 的连锁

  • Drive_reference 删除 → 该 issue 参与者立即失去 file 权限（除非他还能通过其他路径）
  • Sync engine push 通知所有当前看见此 file 的 client → 自动 hide
  • File 已下载到本地的拷贝不可追回（这是物理事实，不能假装能控制）— audit log 记录"who downloaded before revoke"以便事后追溯

8. Audit event schema (drive 维度)

所有 drive 操作发 audit event 到独立 audit 工具（Graylog / OpenSearch）：

{
  "ts":             "2026-05-27T...",
  "operation":      "drive.read|write|move|delete|reference|signed_url",
  "workspace_id":   "...",
  "agent_id":       "...",          // 可空 — 不是 agent 调用时
  "agent_owner":    "...",          // 可空
  "caller_user_id": "...",
  "file_id":        "...",
  "file_size_bytes": 123456,
  "space_id":       "...",
  "space_kind":     "workspace|team|private|agent_output",
  "team_id":        "...",          // 可空 — kind=team 时
  "outcome":        "success|denied|halted|error",
  "intrusive_flags": ["..."],
  "reference_target_type": "...",   // 可空
  "reference_target_id":   "...",   // 可空
  "judge_verdict":  "..."           // 可空 — Layer 2 judge 时
}

消费端：

  • Per-space dashboard：space owner 看见这个 space 这周被访问 N 次，谁访问了什么
  • Per-agent dashboard：主设计 §6 的 agent dashboard 自动包含 drive maturity（这个 agent 读了多少 file，halted 多少次）
  • File-level burn-down：一个文件被谁下载过（点 file 看 access log）
