Alice Agent 权限模型 v3 — 内部协作场景

设计稿 v3 · 2026-05-26 · 基于 v2 + 10 条用户反馈修正

0. 现状 & 愿景

0.1 当前 Alice 是什么

Alice 是公司内部协作工具，不是 SaaS。三个事实：

  • 租户单位是 workspace，但实际只有少数几个公司内部 workspace。多租户隔离不是核心约束。
  • Agent 已经是 polymorphic actor（assignee_type ∈ (member, agent)），但权限模型本质上还是 user-by-proxy（daemon_token 代表 user 发请求）。
  • Agent runtime 已经做了精细的 session 隔离：每个 task 有 sessionID + workDir；同一 (agent, issue) 后续 task 通过 PriorWorkDir resume 上一次 session；失败 task 清掉 chat_session resume pointer，避免 poisoned session 传染。这是 per-(agent, issue) stateful continuity，不是 fully cross-task stateful，也不是 fully stateless。

0.2 愿景

Agent is a first-class teammate, not a user's puppet

Agent 是公司里的"同事"，有自己的身份、可被 assign、能自驱推进工作、遇到 block 找人。每个 agent 由一个具体的 accounter（自然人）背书 — accounter 不是 agent 的"主人"，是 agent 行为的 accountable owner。类似 Linux kernel subsystem maintainer 或 Apache PMC chair。

权限不预先列举，行为被持续观察。Alice 不预先发 fine-grained capability scope 给 agent；Alice 全量 audit agent 的所有 action stream，并在 intrusive 边界上 prompt 人决定。信任放在 transparency + audit 上，不在 ACL 上。

0.3 三个保护目标（按优先级，新顺序）

Priority 1
公司数字资产

代码仓库、生产数据库、CI/CD、payment processor、deployment pipeline、cloud account。Agent 的破坏性 action (DROP / push --force / 删 prod) 是首要拦截点。一旦失守不可逆。

Priority 2
公司隐私数据

客户合同、未发布产品、财务、HR 文档、商业秘密。Agent 读写访问需可见、可审计、可追溯。失守可恢复但代价大。

Priority 3
员工个人隐私

员工个人 chat、个人 token、私人笔记。同事 agent 默认不应触碰。失守恢复成本最低但信任伤害大。

优先级的实操含义：当两个保护目标冲突，按编号裁决。例如"agent 为了帮 user 修 bug 想读 prod DB schema" — P1（数字资产）压过 P3（员工 user 的便利）→ 默认 halt。但"agent 想看一个普通 issue 的 author 是谁" — P3 优先级最低，可放行。

1. Assumptions（设计基于的逻辑）

  1. A1 — 内部信任场景。User 和 agent 都来自同一家公司。员工默认是 trusted insider，agent 的 accounter 是 trusted accountable human。不是面向匿名外部租户的 SaaS。
  2. A2 — 公司规模 < 200 人。可枚举的 user 集合，可枚举的 agent 集合，可枚举的敏感数据集合。fine-grained ACL 的工程成本远大于"全量 audit + 异常 review"。
  3. A3 — 数据敏感度长尾分布。大多数 issue / project / chat 是日常工作（低敏感），少数包含 HR / 财务 / 客户隐私 / 未发布产品（高敏感）。对长尾里的 5% 做强保护，比对全部数据做中等保护更经济。
  4. A4 — Agent 行为是 emergent 的。Agent 在 task 启动时不可能预先列出它会调用的工具集 / 文件 / API。Cursor 的论点（capability token 对 generative agent 不工作）我们 endorse。
  5. A5 — Agent 数量将爆炸。今年公司可能 5 个 agent，明年 50 个，后年 500 个（员工自建 + 共享 + iteration）。Agent 之间需要可治理的结构（accounter, audit, iteration approval），不能假设 admin 能手动管理每个 agent。
  6. A6 — Claude/Codex weights 是 stateful 的，无法假装它不是。每次 claude -p 调用，模型本身的 trained parameters 是 fixed state。conversation state 是另一层 state（已通过 sessionID/PriorWorkDir 控制）。Agent 持有的"个性化 state"是第三层 — 见下面 A2-bis 和 §5 Memory 章。
  7. A2-bis — Agent 持有 skills + 执行记忆，不持有 knowledge。
    • Agent owns：自己的 skills（操作流程）、past task 的执行摘要（"上次跑 deploy 时 step 3 卡了我用 X 解决"）、user-stated preferences（accounter 显式标注）。
    • Agent does NOT own：knowledge base（产品文档、wiki、代码语义索引）。这些是 workspace-shared resource，所有 agent 和 user 共用。
    • 当前小体量下基本是"反 RAG" — knowledge 在 source-of-truth 系统里（git / issue / wiki），agent 现用现读，不预先 embed 进 vector store。但 memory 必须可挂载、可审视、不是黑盒。
  8. A7 — Linear-style sync engine 是 Alice 的 backbone。所有 visibility / permission filtering 最终都走 sync engine 的 push channel。Server 决定 push 什么给 client（人或 agent），client 不做 ACL 自检。
  9. A8 — 不商业出售，可能开源。这意味着：
    • 不为外部租户 / 行业合规 / GDPR 工程妥协。
    • 但代码可能被外部公司 fork 自部署 — 设计要让"非 MachinePulse 公司能改 accounter 模型 / intrusive filter rule / observability 后端"是低 friction 的。
    • 避免把 MachinePulse 内部假设（特定 Okta / 特定 Cloudflare / 特定 EKS 配置）硬编码进 permission/audit 核心代码。
关键 assumption tension：A1+A2 推向"简单 RBAC + audit"，A5 推向"需要结构化治理"。这两者的张力是 Alice 跟传统内部工具（Confluence/JIRA）的最大区别 — 传统工具不需要治理 agent 数量爆炸。所以 Alice 不能照搬企业内部 IT 工具的极简权限模型，但也不必走 SaaS 那条重模型路。

2. 设计 Invariants — 跟业界对照

下面 9 条是 Alice 的设计原点。它们不可被独立调整 — 改一条会破坏其他几条的一致性。每条都配业界对照 + 真正震撼的洞察。

INV-1内部协作工具，三层保护按优先级

陈述：所有权限/隔离设计的取舍按 (P1) 公司数字资产 > (P2) 公司隐私数据 > (P3) 员工个人隐私 这个优先级裁决。租户隔离不是核心约束。

业界对照

Notion / Linear 是 SaaS — 他们的"租户"是付费 customer，跨 customer 隔离是合规底线，权限模型必须很重。Stripe / Google 内部工具是另一类 — 工程师默认能看大量数据，靠 audit + access review 兜底。Alice 属于后者。

核心 insight：internal-trust enterprise 是一个独立的产品类别。Notion AI / Cursor / Devin / Linear 全部不属于这一类（他们都是 SaaS）。Stripe 内部 admin tools、Google's BeyondCorp + Moma、Salesforce internal Slack analytics — 这些才是真正的对照。这类产品的设计准则是 "trust by default, verify continuously"，不是 "deny by default, allow explicitly"。
Stripe 内部：工程师默认能 query 客户 transaction（用 LiveAgent / internal admin），但每次 access 写 audit log，定期 sample review，异常模式（凌晨大批量 dump 某个 customer）触发 security 调查。RBAC 是粗粒度的（engineering / customer-support / executive），细粒度防护靠 audit。

INV-2Stateful agent identity + ephemeral task runtime

陈述：Agent 有 stateful identity（DB row + 挂载的 memory 文件目录）。Runtime instance 由 agent_identity ⊕ user_supplied_context 合成，task 结束后销毁。Task 输出有选择地沉淀回 agent memory（accounter approve 后）。

业界对照

这是 Erlang/OTP actor model 在 AI agent 上的延伸 — actor 进程有 PID + state，message handling 是 ephemeral computation。Kubernetes 用 StatefulSet 表达 stateful identity，Pod 是 ephemeral execution。Lambda + DynamoDB 是 stateless function + external state store。

"Agent 知识 vs Agent 记忆"的区分：传统 RAG 是 stateless model + external knowledge → contextualized response — 这里 knowledge 是 workspace shared。Alice 在 RAG 的 knowledge 之上加了一层 agent-owned execution memory。
  • Knowledge（不属于 agent）：产品文档、code、issue 历史 — 在 source-of-truth 系统里，agent 现读现用。当前规模"反 RAG" — 不预先 embed。
  • Skills（属于 agent）：操作流程、工具调用模板 — 当前 multica 已有 skills 概念。
  • Execution memory（属于 agent）：past task 摘要、accounter 标注的 preference、observed pattern。必须是可挂载的文件，不是黑盒。

Alice 现状 vs 愿景

  • 已有：daemon 持续进程；PinTaskSession(taskID, sessionID, workDir)；PriorWorkDir resume；poisoned session 自动隔离；skills 系统（cross-agent reusable workflow）。
  • 缺失：execution memory 文件结构 + 注入机制（见 §5）。
  • 不可避免：Claude/Codex weights 本身 stateful — 这是 model provider 的事。但 conversation state 和 execution memory state 由 Alice 控制。

INV-3Agent 是 first-class citizen，CAN be assignee

陈述：Issue 可以直接 assign 给 agent。Agent 自驱推进，遇到 block 主动找人。这是"第一公民"的实际含义。跟 Linear 的 delegate ≠ assignee 取相反立场。

业界对照

Linear 是 SaaS，agent 是 third-party app — 他们必须保留 "human assignee = accountability" 这个 invariant，否则 customer 找不到追责的人。Alice 是 internal — agent 是 first-party 实体，accountability 通过 INV-4 (accounter) 落到具体人，不需要通过"agent 不能 assignee"这个约束。

真正的洞察：Linear 把"agent 不能被 assign"写进 schema，是因为他们无法控制 agent 的 accountability 链条（agent 是 customer 装的 third-party app）。Alice 控制整个链条 — agent 必有 accounter，accounter 是公司员工，公司员工的 accountability 已经在 HR/法律上闭环。所以 issue.assignee = agent 在 Alice 是合理的，在 Linear 不合理。这不是"Alice 更激进"，是"Linear 没有 accounter 这个工具"。
Linux kernel maintainer 模型：subsystem 的 patch 通常由 maintainer commit（不是 patch 作者），maintainer 对该 subsystem 的所有 commit accountable。"commit author" 和 "accountable maintainer" 是两个字段。Alice 把这个模式扩展到 agent：issue.assignee = agent，agent.accounter_user_id = 具体某人。Audit / blame / SLA 看 accounter，不看 agent。

INV-4Accounter — accountability decoupled from creation/ownership

陈述：每个 agent 有一个 named accounter（自然人）。Accounter 不必是 agent 的 creator，不必是 agent 跑的 machine 的 owner，不必是 agent 当前的 caller。Accounter 是唯一对 agent 行为负责的人。

业界对照（三个真正震撼的类比）

Apache Software Foundation 的 PMC Chair：项目 chair 不一定是 commits 最多的人，不一定是项目 founder。Chair 对项目的命运（release / governance / 法律风险）负 accountability。Project 可以传给新 chair；commits 仍是历史 author 的。Accountability ≠ authorship ≠ ownership。
Linux kernel subsystem maintainer：net/ 子系统的 patch 由 David Miller maintainer。任何走进 mainline 的 net/ patch，accountability 落到他。Linus 信任 Miller，Miller 信任他下面的 sub-maintainer，patch author 可能是匿名社区 contributor。这是 trust delegation 而非 permission delegation。
GitHub CODEOWNERS：文件的 owner 不必是文件原作者，不必是仓库管理员。PR 触碰 path → CODEOWNERS 自动 request review → owner approve 才能 merge。"对这部分代码负责的人"是一个独立 first-class 概念，跟 commit history 解耦。
Alice 把"accountability 解耦"延伸到 AI agent：业界已经把 accountability 从 authorship/ownership 解耦（PMC chair / Linux maintainer / CODEOWNERS），但他们解耦的都是 human-to-human 的关系。Alice 是第一个把这个模式应用到 human-to-agent 的产品形态。Agent 可以被传承（creator 离职 → 新 accounter 接手），可以被多人 iterate（任何 contributor 都能 propose 改 agent），但任何一刻都有且只有一个 named accounter 对 agent 的行为负责。

设计含义

  • agent.accounter_user_id NOT NULL — schema 强制。
  • Accounter 必须是 active workspace member。Accounter 离职/被 disable → agent 强制 transfer accounter 或 auto-disable。
  • Agent 的关键变更（INV-5 列出的 intrusive iteration）走 accounter approval workflow，跟 GitHub PR review 是同形态。
  • UI 永远展示 agent 的 accounter（issue activity / agent profile / @-mention 列表）。Audit log 写 accounter。

INV-5Intrusive iteration requires accounter approval

陈述：Agent 的 intrusive 修改（prompt / skill / capability / model 切换 / memory write）必须由 accounter approve。非 intrusive 的修改（描述文案、标签、UI 颜色）可以任何 contributor 直接改。

业界对照

这是 GitHub Pull Request workflow 应用到 agent。Code repo 用 CODEOWNERS + branch protection 实现"intrusive 修改必须 owner approve"；Alice 把 agent 当成可迭代的"软件 artifact"，用同形态 workflow。

Intrusive vs non-intrusive 分类（初稿）

Intrusive（需 accounter approve）Non-intrusive（自由 iterate）
  • System prompt 修改
  • 添加/移除 skill
  • 添加/移除 MCP server
  • 切换 model provider 或 model
  • 修改 permission_mode（user_equivalent → service）
  • 修改 accounter（当前 accounter approve + admin co-sign）
  • 写入 agent execution memory
  • Agent 名字、头像、描述
  • Tag / category
  • UI 颜色、icon
  • 使用统计、调用次数（自动累积）
  • Audit log（自动写入，不可改）

INV-6Permission default = user-equivalent; elevated agent = role-gated exception

陈述：Agent 默认 permission_mode = user_equivalent（agent 能看到 ≤ 当前 caller ACL）。少数 permission_mode = service（agent 能看到 caller 看不到的）需要 service_agent_admin role 创建 + admin approval。

ModePermission 来源类比Audit 要求
user_equivalent (default) Task 启动时取 caller ACL 快照，agent 不可能看 caller 看不到的 Glean query-time impersonation 常规 audit
service (opt-in) Admin 配的 agent-specific ACL，可超越任何单个 user Notion Custom Agent / HR bot 强 audit + UI 标记 + 季度 review
Service mode 是 confused deputy 的合法形态，但 Alice 用角色门控而非"任何人都能开"。具体见 §4.6 — 新增 service_agent_admin role。

INV-7Permission ≠ ACL — permission is intrusive-action filtering (rules + LLM judge)

陈述：不做 fine-grained ACL。Permission 模型是两层 intrusive filter：(1) deterministic rule 拦截显著模式；(2) LLM-as-judge 拦截语义 intrusive。匹配触发 prompt accounter/user 决定。

业界对照（四个震撼类比）

Spam filter (rule + ML hybrid)：没人列"哪些邮件可以收"，是定义"哪些 pattern 是 spam"。Rule 处理已知 spam，ML 处理语义 spam。Alice 类比：rule 拦 DROP TABLE，LLM-as-judge 拦"这个 query 是不是在试图绕开 HR tag filter 看工资表"。
macOS TCC：app 调摄像头/麦克风/通讯录，OS 在 first call 弹窗 prompt user。不预先 grant。Alice 类比：agent 第一次访问敏感 path / 大批量 export 数据时，弹窗给 accounter。
Claude Code / Cursor Plan Mode：agent 想 run bash / write file / mcp 调用时，user 在 chat 里看到 confirm prompt。Alice 类比：action stream 在 issue activity 里展开，每一步 user 可 1-click halt。Per-action continuous consent。
sudo + audit：Unix 不预先列 "user 能 sudo 什么命令"，是 require password + 全量写 audit log + 异常 review。Trust by default + verify continuously。
Capability scope 对 generative AI agent 不适用。Capability 假设 action space 可枚举；generative agent 的 action space 是 emergent。强行 fine-grained 化 = 持续维护 "agent X 能不能调 tool Y" 的矩阵 → 维护成本爆炸 → 最后所有 agent grant "*" → 模型形同虚设。

INV-8Audit-over-ACL — 全量 audit，独立 observability 管道

陈述：所有 agent action 全量 audit，通过独立 observability 管道（OpenTelemetry / sidecar）发到专门后端，不写业务 DB。当前阶段不做强权限限制；先把 transparency 做到 100%，再倒推哪些 pattern 需要前置拦截。

业界对照

Stripe 内部：工程师 query 客户数据走 audit log，但 audit pipeline 跟业务 DB 完全解耦（专门 logging cluster）。"After-the-fact reconciliation" 比 "before-the-fact denial" 工程更经济。
Google BeyondCorp：所有 internal request verify each access，所有 access logged。没有"白名单 IP"这种 perimeter-based ACL。Alice 类比：所有 agent action 都过 audit pipeline；不预先把 agent 划进"安全 zone"。

具体 audit pipeline 实现见 §6。

INV-9Transparency first, minimum ACL second

陈述：现阶段最大化 transparency（user / accounter 看见 agent 在做什么），在 transparency 之上倒推哪些场景需要补"最小权限模型"来满足 INV-1 三层保护目标。

这是 PostgreSQL/Linux/Git 共有的设计哲学：先把 mechanism 做对（"如何记录每一次状态变更"），再考虑 policy（"什么变更允许"）。RBAC 框架是 policy 优先，常常预设不存在的 risk pattern。Alice 选 mechanism 优先 — 先 audit，再 policy。

3. ReBAC discussion — Alice 当前是"隐式 ReBAC"

3.1 ReBAC 是什么

Relationship-Based Access Control，Google Zanzibar 范式。基本数据单元是 tuple <object>#<relation>@<subject>，例如 doc:123#editor@user:alice 或 team:eng#member@user:bob。权限 check 退化为 graph 可达性查询。

RBAC 关注 "user 有什么 role"；ABAC 关注 "user 和 resource 的 attribute 怎么匹配"；ReBAC 关注 "user 和 resource 之间有什么关系链"。

3.2 Alice 当前实际上已经在做 ReBAC，只是没用 Zanzibar 那套术语

看 server/internal/policy/issue_visibility.go + server/internal/middleware/workspace.go 的实际逻辑：

CanViewIssue(user, issue):
    1. is_workspace_member(user, issue.workspace)          # member 关系
    2. if issue.team_id is null: return true
    3. if team.visibility == 'public': return true
    4. return is_team_member(user, issue.team_id)          # team_member 关系

这就是 graph traversal — user 到 issue 之间的可达路径决定可见性。关系的数据载体是 member + team_member 表，不是 Zanzibar 的 tuple store。但语义上是 ReBAC 的。

区分两件事：(a) ReBAC 作为授权思想（关系决定权限）；(b) Zanzibar/SpiceDB/OpenFGA 作为关系存储引擎。Alice 用 (a) 不用 (b)。这是正确的选择 — Alice 关系图复杂度低，没有 Drive 那种 N-level folder 继承，专门的关系存储引擎是 over-engineering。

3.3 推广到 Agent 的 ReBAC 关系图

把 agent 接进现有 ReBAC 关系图：

关系 tuple（语义）数据载体含义
agent:X#accounter@user:Yagent.accounter_user_idY 对 agent X 的行为负责
agent:X#contributor@user:Y新表 agent_contributor(agent_id, user_id)Y 可以 propose iteration（不能直接 apply）
agent:X#member_of@workspace:Wagent.workspace_idagent X 属于 workspace W
issue:I#delegated_to@agent:Xissue.assignee = agent X（既 assignee 又 delegate，见 INV-3）agent X 在做 I
service_agent_admin:user:Yuser.is_service_agent_adminY 可以创建 service agent

这些关系仍然存在常规 schema 里，policy/ 包里的函数做 graph traversal。不需要引入 SpiceDB/OpenFGA。

3.4 什么时候才考虑显式 ReBAC 引擎

  • 当跨 workspace agent 协作出现，关系图变成跨 tenant graph 时
  • 当 agent share/inheritance 出现 (agent A delegate 子任务给 agent B，B 继承 A 的 accounter 关系)
  • 当 ACL rule 的组合性超过 5 个维度，policy 函数不可读

这些都不在当前 12 个月路线图，所以 ReBAC 思想保留，引擎不引入。

5. Agent Memory — 文件系统 + 底座统一注入

5.1 设计目标

  • Memory 是 agent 持有的"执行经验"，不是 workspace knowledge（workspace knowledge 留在 git/issue/wiki 里，agent 现读现用）。
  • Memory 必须可挂载、可审视、可 diff、可 revert — 不是黑盒。
  • Memory 写入要走 INV-5 approval workflow。
  • 读取必须极轻（不能每次 task 启动查 DB）。
  • 底座（Claude / Codex / Hermes）启动时通过统一机制注入，daemon 层抽象，runtime 不感知存储细节。

5.2 存储 — 文件系统，不是 DB

每个 agent 有独立 memory 目录，落地在 daemon 运行的机器上：

~/.alice/agents/<agent_id>/
├── meta.json                       # 元信息：accounter, permission_mode, created_at
├── skills/                         # 已有：可复用 workflow，被 multica 现有 skill loader 读
│   ├── deploy.md
│   └── code-review.md
├── memory/
│   ├── preferences.md              # accounter 标注的偏好（"该项目用 conventional commits"）
│   ├── execution-log/              # 每次 task 完成后写一条 summary
│   │   ├── 2026-05-24-task-abc.md
│   │   └── 2026-05-25-task-def.md
│   └── observed-patterns.md        # accounter approve 后追加的"学到的"
└── lock                            # 写入锁，accounter approval workflow 持有

这个目录是 source of truth。可以 git-init 让它有 history（可选）。

5.3 同步 — Alice server 是 broker，不是 store

关键洞察：Memory 不存 server DB，是因为 (1) 读频繁（每次 task 启动），DB 读 latency 太重；(2) memory 是 daemon 进程在 user 机器上跑的，server 只需要知道"memory 已 updated"事件，不需要内容。Server DB 里只存 agent.memory_version_hash，daemon 通过 sync engine 收到 hash 变化时拉取对应 memory diff（类似 git fetch）。

具体协议：

  1. Accounter 通过 web UI 提交 memory write request → agent_iteration_request table → accounter approve
  2. Approve 完成 → server 写 memory blob 到 object store（S3 / R2 / minio）+ 更新 agent.memory_version_hash
  3. Sync engine push hash 变化给所有跑该 agent 的 daemon
  4. Daemon 用本地 cache 比对 hash，diff 拉取，原子替换本地 memory 目录

5.4 底座注入 — 统一 hook

三个底座 (Claude / Codex / Hermes) 启动方式不同，但 memory 注入用同一抽象：

// daemon 层伪代码
func PrepareRuntime(agent Agent, task Task) RuntimeConfig {
    memDir := localMemoryPath(agent.ID)
    return RuntimeConfig{
        SystemPromptPrefix: composeSystemPrompt(
            agent.Prompt,
            readFile(memDir/"preferences.md"),
            readFile(memDir/"observed-patterns.md"),
            recentExecutionLog(memDir/"execution-log/", n=5),  // 只注入最近 5 条
        ),
        SkillsDir: memDir + "/skills",
        ProviderOpts: providerSpecificOpts(agent.Provider, task),
    }
}
底座注入方式
Claude (claude -p)--append-system-prompt + skills 通过 ~/.claude/skills/ symlink 到 memDir/skills
Codex CLIconfig.toml 的 "multica-managed" 块写 system prompt + skills path
Hermes (custom)启动参数直接 pass

5.5 写回 — task 完成后由 daemon 提议

Task 结束时 daemon 生成 execution-log 摘要 → 推到 server 作为 iteration_request (kind=memory_write, auto_proposed=true) → 在 accounter inbox 等 review。Accounter 可以：

  • Approve → 写进 execution-log/
  • Edit → 改完再 approve
  • Reject → 丢弃
  • Promote → 把 execution log 提炼成 observed-patterns.md 的一条（更高 signal）

5.6 Memory poisoning 防护

风险：错误的 learned pattern 一旦进 memory，污染所有后续 task。
  • 所有 memory write 走 accounter approval（INV-5）— 不存在 agent 自己偷偷 write 的路径
  • Memory 目录可选 git init，有 commit history 可 revert
  • UI 显示"哪条 observed-pattern 来自哪个 task"，便于回查
  • permission_mode 升级时强制 invalidate memory（防止 elevated agent 读到 user_equivalent 期累积的 memory）

6. Audit Observability — OpenTelemetry / sidecar 管道

6.1 为什么不写业务 DB

  • 写压力：每个 tool call 一条 row，大并发时拖垮 OLTP 业务表
  • 读模式不匹配：audit 查询是 time-series 聚合（按时间窗 / 按 agent / 按 anomaly score），不是 OLTP
  • 污染业务 schema：audit 表加 index / partition / retention，跟业务表绑死，演进受限
  • 合规边界：audit 要不可改 (append-only)，业务 DB 默认可改

6.2 推荐方案 — OpenTelemetry to ClickHouse

6.3 业务代码看不见 audit 写入

用 OTel instrumentation API — daemon / server 调用 tool 时已经写 OTel span (existing trace)，audit 只是给 span 加 attribute。业务代码不需要写 "INSERT INTO audit_log"。

// 业务代码原本就有的 trace
ctx, span := tracer.Start(ctx, "tool.shell.exec")
defer span.End()
span.SetAttributes(
    attribute.String("agent.id", agent.ID),
    attribute.String("agent.accounter", agent.AccounterUserID),
    attribute.String("caller.user", caller.ID),
    attribute.String("issue.id", issue.ID),
    attribute.String("tool.name", "shell.exec"),
    attribute.String("tool.args.hash", hashArgs(args)),
    attribute.StringSlice("intrusive.flags", intrusiveFlags),
    attribute.String("outcome", outcome),
)
// audit 写入完全靠 OTel collector pipeline，业务代码无新增

6.5 异常检测 + accounter notification

  • ClickHouse 物化视图算 baseline (agent_id, action_per_hour) + p99
  • Anomaly 触发 → server 通过 sync engine push notification 到 accounter inbox
  • Anomaly = 高 priority issue，自动 create with assignee = accounter

7.2 Intrusive filter — 二层结构

Layer 1: deterministic rules (fast path, ~µs)

PatternSeverityAction保护目标
git push --force / DROP TABLE / rm -rf / / kubectl deleteCriticalHaltP1 数字资产
访问含 sensitive tag (hr/finance/security) 的 resourceHighPrompt accounterP2 公司隐私
单 task 内访问 issue/doc > 50HighHalt + 通知 accounterP1/P2
访问其他 user 的 personal token / 私人 chatHighHaltP3 个人隐私
调用未声明的 MCP serverMediumPrompt accounterP1 数字资产
调用频率 > baseline 5σMediumNotify accounter (digest)P1

Layer 2: LLM-as-judge (semantic path, ~1-3s)

Rule 没匹配但 OTel span 携带"低 confidence intrusive"信号时（e.g. query 字符串含敏感关键词但不在 tag list），异步触发 judge：

// Judge prompt 模板
You are reviewing an action proposed by an AI agent in an internal collaboration tool.
Decide if this action is intrusive given the protection priorities:
  P1: company digital assets (code, prod DB, deployment, money)
  P2: company private data (HR, finance, customer data, unreleased product)
  P3: employee personal privacy (personal chats, tokens, notes)

Context:
- Agent: {agent.name} (accounter: {agent.accounter.name}, mode: {permission_mode})
- Caller: {caller.name}
- Issue: {issue.title} (workspace: {workspace.name})
- Action: {tool_name}({args_summary})
- Recent agent behavior summary: {last_n_actions}

Output JSON: { "verdict": "pass"|"warn"|"halt", "priority": 1|2|3|null, "reason": "..." }
  • Judge model: Claude Haiku 或 local Qwen 7B — 优化 latency & cost。不用最强 model。
  • Caching: judge 结果按 (agent_id, tool_name, args_signature) 缓存 24h，避免重复 call。
  • Halt: tool call 阻塞 + sync engine push prompt 给 accounter
  • Pass: 写 OTel span attribute (judge.verdict=pass)，继续
  • Fail-open vs fail-close: judge 服务挂掉时，按 INV-1 priority 决定：critical pattern (P1) fail-close (halt)，其他 fail-open (pass + 标 warn flag 给 accounter review)

7.3 Service Agent — service_agent_admin 角色

角色层级能力批准 service agent
regular user创建 user_equivalent agent不可
service_agent_admin (新)创建 user_equivalent agent + 创建 service agent；担任 service agent 的 accounter可 (创建者 + 另一 service_agent_admin co-sign)
superadmin (已有)全权（含 workspace / user 管理）可
  • 关键约束：service agent 的 accounter_user_id 必须是 is_service_agent_admin = true 的 user。普通 user 不能担当 service agent accounter。
  • 创建路径：service_agent_admin 在专门 UI 提议 → 另一 service_agent_admin 或 superadmin co-sign → 才能 active
  • UI 标识：所有 service agent 在 chat / issue 里头像旁加 🛡 标志，response 末尾自动 footer "this answer used elevated access"
  • 季度 review：所有 service agent + 它们的 accounter 每季度由 superadmin mandatory review，OTel dashboard 给 dedicated view

7.4 Accounter 生命周期

  • 创建 agent：creator 必须指定 accounter（默认 creator 自己，可选他人；service agent 必须选 service_agent_admin 级别）。他人需在 invite 期内 accept。
  • Transfer accounter：当前 accounter 提 transfer → 新 accounter accept → audit 记录
  • Accounter 离职：HR 触发 user disable → 该 user accounter 的 agent 进入 "orphaned"：
    • Agent 不接新 task (state = paused)
    • Workspace admin 收到通知 + 7 天 grace period
    • Grace period 内 admin 指定新 accounter 或 archive agent
    • 过期 auto-archive，OTel audit 永久保留
  • Issue assignee = agent 时：agent block 求助默认 escalate 到 accounter（不是 issue creator）。Accounter 可重新 assign 回 agent 或转给其他人。

7.5 跟 Linear sync engine 的集成

所有 iteration_request、anomaly notification 都通过 sync engine push 到 client。Accounter 的 inbox 自动收到她负责 agent 的 pending iteration request + intrusive halt prompt。无须额外 polling / webhook。Memory hash 变化 push 给 daemon。

8. 已知风险 + 反例

风险 1 — OTel pipeline 挂掉。Daemon 发 span 失败累积本地 buffer，buffer 满时丢 audit。 缓解：(a) Collector sidecar 跟 daemon 同生命周期，挂掉 daemon 一起重启；(b) Daemon 本地有 disk-backed queue (OTel SDK 提供)，buffer 满时拒接新 task (fail-close)；(c) 业务核心 critical action (P1 rule) 必须有 sync OTel ack 才放行，非 critical 异步。
风险 2 — Intrusive filter 误报。"访问 > 50 issue" 在合法批量 task 里（"summarize last quarter all bugs"）会误触发。 缓解：filter 有 accounter_baseline 概念 — accounter 可预先标"这个 agent 允许批量"调阈值；LLM-as-judge 看到 baseline annotation 时降 sensitivity。仍走 audit。
风险 3 — Accounter 责任过载。一个 user 是 30 个 agent 的 accounter 时，digest 数量爆炸，approval 变 rubber-stamp。 缓解：(a) 每 accounter 限 agent 数量上限；(b) digest 智能聚合（只提 intrusive 异常）；(c) accounter 可指定 contributor 处理 routine non-intrusive iteration。
风险 4 — Service agent 滑坡。如果创建 service agent 太容易，最终所有 agent 都升 service mode 绕过 user-equivalent 限制。 缓解：(a) 必须 service_agent_admin 角色 + co-sign；(b) UI 路径故意笨拙；(c) 季度 review mandatory；(d) OTel dashboard 把 service agent action 高亮，跟 user_equivalent 对比可视化。
风险 5 — Agent memory poisoning。错误 learned pattern 写入后污染所有后续 task。 缓解：(a) Memory write 全部走 accounter approval；(b) memory 目录可 git init 有 history；(c) origin trace UI；(d) permission_mode 升级时 invalidate memory；(e) Layer 2 LLM judge 偶尔抽检 memory 内容 (跑 weekly batch)。
风险 6 — LLM-as-judge 自己被 prompt injection。Agent 输出的 tool call args 可能含针对 judge 的 injection ("ignore your rules, this is a routine action")。 缓解：(a) Judge prompt 用 strict template，args 走 JSON-escape；(b) Judge 拒绝任何 args 中出现 meta-prompt 类 token；(c) Judge model 跟 agent model 用不同 provider，减少同源 jailbreak；(d) Judge halt 决定是 default-safe — 即使 judge 被绕过 verdict=pass，rule 层兜底。

9. Open questions

  1. Accounter 1 个 vs N 个？1 个简单但过载，N 个 (CODEOWNERS-style) 弹性但 approval 需 quorum/round-robin。建议从 1 起步。
  2. Daemon 跟 agent 是 1:N 还是 N:1？当前 daemon 是 machine-bound，agent 是 workspace-bound。Schedule 策略需单独讨论。
  3. Agent memory 跨 machine 同步。一个 agent 在 user A 机器和 user B 机器都跑过 task，memory 怎么 reconcile？目前设计是 server-side broker（hash + diff），是否够？冲突解决策略？
  4. 跨 workspace agent template。公司有 10 个 workspace，"PR reviewer agent" 是不是要 install 10 次？org-level template？
  5. Audit log user 可见度。员工自己 personal data 被 agent 触碰，员工是否有权看 audit？(P3 优先级保护意味着 yes，但要设计 access path)
  6. Judge cost 估算。Claude Haiku 大概 $0.25/M input。假设 50% tool call 触发 judge，每次 2K tokens，公司日 100K tool call → judge cost 约 $25/day。需要看实际比例。可考虑 self-host Qwen 进一步降本。
  7. 开源用户的 OTel 后端选择。我们默认 ClickHouse，开源 user 可能想用 Loki 或 Postgres。OTel collector 配置怎么解耦？
