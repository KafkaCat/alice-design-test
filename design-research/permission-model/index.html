Alice 权限模型 · 核心不变量

内部协作 + AI agent 第一公民 · 设计圣经（bible）。每条 = 一条可裁决的准则；比较两个设计方案时，逐条对照取最终判决。只写理想态——落地差距见系统级清单。

Agent 是公司的「同事」，不是某个 user 的提线木偶——有身份、能被 assign、自驱推进、遇 block 找人。

每个 agent / machine = 一个 owner（责任主体）+ N 个 collaborator（只协作）。信任建在 transparency + audit 上，不建在 fine-grained ACL 上。

§0 · 三个保护目标 = 最高裁决准则

Priority 1
公司数字资产

代码 / 生产 DB / CI·CD / cloud account。破坏性 action 是首要拦截点。失守不可逆。

Priority 2
公司隐私数据

客户合同 / 未发布产品 / 财务 / HR。读写须可见可审计。失守可恢复但代价大。

Priority 3
员工个人隐私

个人 chat / token / 私人笔记。同事 agent 默认不碰。信任伤害大。

⚖ 总裁决：任意两个目标 / 两条法条 / 两个方案冲突时，编号小的赢。例：agent 为便利想读 prod DB —— P3（便利）撞 P1（资产）→ 默认 halt。

前提（撑起全部法条）：内部信任，非匿名 SaaS · 规模 <200 人，集合可枚举 · agent 行为 emergent，capability 列不全 · agent 数量将爆炸，必须可治理 · 可能开源，audit/规则后端要可替换。

§1 · 地基

INV-1内部信任：trust by default, verify continuously

取舍按 P1 > P2 > P3；准则是「默认信任 + 持续验证」，不是「默认拒绝 + 显式放行」。租户隔离不是核心约束。

方案偏「事前锁死 / deny-by-default 重 ACL」vs 偏「放行 + 全量 audit + 异常 review」→ 选后者。（对照系是 Stripe 内部 / BeyondCorp，不是 Notion / Linear 这类 SaaS。）
INV-2Workspace 是强边界（hard wall）

跨 workspace 默认完全不可达——user / agent / machine / drive / issue / memory 一律如此。

任何「让跨 workspace 可达」的方案直接否决。要隔离敏感数据（HR / 财务）→ 选「拆独立 workspace」，不选「workspace 内加 file-level ACL」。

§2 · 协作核心

INV-3Agent 是第一公民

Agent 跟 user 同等地位——能被 assign（issue.assignee = agent）、能 own、能 collaborate。

设计「agent 能不能做 X」时，默认给它 user 同等权，除非更高法条挡住。（Linear 做不到是因为它没有 owner 这工具；Alice 有，所以可以。）
INV-4一个 Owner + N 个 Collaborator（agent 和 machine 都适用）

每个 agent / machine 有且仅一个 owner（全权）+ 任意collaborator（只协作）。这是所有资源权限的唯一形状。

操作OwnerCollaborator
Agent：assign task / 调用 / 观察 action stream✅✅
Agent：改 prompt / skill / MCP / model / permission_mode✅✘
Agent：删除 / 转移 owner / 增删 collaborator✅✘
Machine：跑自己的 agent / 看状态✅✅
Machine：改配置 / 改 visibility / 下线 / 管 collaborator✅✘
任何资源权限设计必须落到这个形状。否决「无主资源」（没有唯一责任人）和「人人可改基线」（collaborator 能改 owner-level 配置）。

Owner ≠ creator ≠ caller ≠ 机器拥有者：owner 是当前责任人，可 transfer（creator 不变），可在 A 部门却让 agent 跑 B 的机器、被 C 调用。业界先例：Apache PMC chair / Linux maintainer / GitHub CODEOWNERS——accountability 早已跟 authorship 解耦，Alice 把它用到 human-to-agent。

INV-5Team 是组织视图，不是加密边界

Team 决定 workspace 内部「谁看哪些 issue / project」（public 全员可见 / private 仅成员）；issue 绑恰好一个 team。

需要「强隔离 / 加密」→ 用 workspace（INV-2），不用 team。Agent 跟 user 一样守 team 边界，不是例外（assign 到 private team 需显式 access）。
INV-6两条正交权限轴

系统角色（member / admin / superadmin）⊥ 资源关系（owner / collaborator）——独立判断，互不蕴含。

否决任何把两轴耦合的方案。典型反例：「admin 自动成为所有 agent 的 owner」——admin 有管理权 ≠ 对每个资源有责任关系。

§3 · 权限机制（mechanism over policy）

INV-7Permission ≠ ACL

不做 fine-grained capability / ACL 矩阵；权限 = 两层 intrusive filter（rule 拦显著模式 + LLM-judge 拦语义 intrusive），命中即 prompt owner/user。

方案 PK：「加 capability scope / ACL 矩阵」vs「加 filter + audit」→ 永远选后者。理由：generative agent 的 action space 不可枚举，矩阵必腐烂成全员 grant *。（垃圾邮件 / macOS TCC / sudo 都是「拦异常」而非「列白名单」。）
INV-8默认 user-equivalent；service = admin 门控的例外

Agent 看到的 ⊆ caller 看到的（取 caller ACL 快照）。要越权（service mode）必须 admin 创建 + co-sign + 强 audit + 季度 review。

默认拒绝 elevated。service mode 创建路径要故意笨拙，防止所有 agent 都升 service 绕开限制。
INV-9改基线需 owner approve

改 agent 行为基线（prompt / skill / MCP / model / permission_mode / 转 owner）走 owner approval；非基线（名字 / tag / UI）和日常 memory 写入（模型自决定）不打扰 owner。

判断「这变更要不要 approval」就问一句：它改变 agent 的行为吗？ 改 → approve；不改 → 放行。（同 GitHub PR + CODEOWNERS 形态。）
INV-10Audit-over-ACL

全量 audit，用专用开源 audit 工具（Graylog / OpenSearch，append-only、长期保留、带 UI），跟应用观测（APM / metrics）完全分离。

方案 PK：「before-the-fact denial」vs「after-the-fact audit」→ 选 audit（更经济）。先把 transparency 做到 100%，再倒推哪些 pattern 要前置拦截。审计后端别绑死，留给 fork 替换。

§4 · 状态

INV-11状态三层，各归其位

Identity stateful（DB row + 挂载 memory）· Runtime ephemeral（task 后销毁）· Memory 能力在底座（11 种各自写），Alice 只做最上层 pointer + metadata 索引，不集中存内容。

设计任何「agent 记住 / 学到」的东西：内容放底座，Alice 只存「在哪台 machine / 哪个底座 / 多大 / 何时改」。否决「Alice 重写一套 memory」（必跟底座 schema 打架）。Knowledge（文档/code）不属于 agent，现读现用。

§5 · 关系图速查

关系 tuple语义
workspace:W#member@user:UU 是 W 成员（hard wall 准入）
team:T#member@user:U / @agent:AU / A 进 team T（workspace 内细分可见性）
issue:I#in_team@team:Tissue 属于恰好一个 team
issue:I#assignee@agent:AA 在做 I（第一公民 assignee）
agent:A#owner@user:U / machine:M#owner@user:U唯一责任人
agent:A#collaborator@user:U / machine:M#collaborator@user:U协作者（多个）

权限 check = 这张图上的可达性查询（轻量 ReBAC，关系直接建在业务表，不引入 SpiceDB / OpenFGA）。一条典型判断：

CanView(subject, issue):                 // subject = user 或 agent
    same_workspace(subject, issue)  ?  继续 : 拒绝     // INV-2 hard wall
    issue.team_id is null           ?  允许（workspace member）
    team.visibility == 'public'     ?  允许
    is_team_member(subject, team)                      // INV-5
