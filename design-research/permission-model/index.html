Alice 权限模型 · 核心不变量

内部协作 + AI agent 第一公民 · 这是设计原点（bible）。每条法条相互咬合，改一条会破坏其他几条的一致性。本文只写理想态——落地进度与现状差距见「系统级功能」清单。

Agent 是公司的「同事」，不是某个 user 的提线木偶。它有自己的身份、能被 assign、自驱推进工作、遇到 block 主动找人。

每个 agent（和每台 machine）由一个 owner 背书 + 若干 collaborator 协作。Owner 是责任主体，collaborator 只协作。

权限不预先列举，行为被持续观察。信任建在 transparency + audit 上，不建在 fine-grained ACL 上。

§0 · 三个保护目标（按优先级裁决）

当两个目标冲突，按编号裁决。这是所有取舍的最高准绳。

Priority 1
公司数字资产

代码仓库、生产数据库、CI/CD、payment、deploy pipeline、cloud account。Agent 的破坏性 action（DROP / push --force / 删 prod）是首要拦截点。失守不可逆。

Priority 2
公司隐私数据

客户合同、未发布产品、财务、HR 文档、商业秘密。读写需可见、可审计、可追溯。失守可恢复但代价大。

Priority 3
员工个人隐私

员工个人 chat、个人 token、私人笔记。同事 agent 默认不应触碰。失守恢复成本最低，但信任伤害大。

设计前提（这些假设撑起下面所有法条）
  • 内部信任：user 和 agent 都来自同一家公司，员工是 trusted insider，agent 的 owner 是 trusted accountable human。不是面向匿名租户的 SaaS。
  • 规模可枚举：<200 人，agent / 敏感数据集合都可枚举。fine-grained ACL 的工程成本 ≫「全量 audit + 异常 review」。
  • Agent 行为是 emergent 的：task 启动时无法预先列出它会调用的工具/文件/API。所以 capability scope 这条路对 generative agent 不通。
  • Agent 数量将爆炸：今年 5 个，明年 50，后年 500。必须有可治理的结构（owner / collaborator / audit），不能假设 admin 手动管每个 agent。
  • 不商业出售，可能开源：不为外部合规妥协；但 audit 后端 / owner 规则 / filter 规则要让外部 fork 能低成本替换。

§1 · 地基公理

INV-1内部协作工具 — trust by default, verify continuously

陈述：所有权限/隔离取舍按 (P1) 数字资产 > (P2) 隐私数据 > (P3) 个人隐私 裁决。设计准则是「默认信任 + 持续验证」，不是「默认拒绝 + 显式放行」。租户隔离不是核心约束。

Alice 属于 internal-trust enterprise 这个独立产品类别——Notion / Linear / Cursor / Devin / Glean 全是 SaaS，跨租户隔离是它们的合规底线，权限模型必须重。Alice 的对照是 Stripe 内部 admin、Google BeyondCorp，这类靠 audit 兜底而非 ACL 锁死。

Stripe 内部：工程师默认能 query 客户数据，但每次 access 写 audit log、定期 sample review、异常模式触发调查。RBAC 是粗粒度的，细粒度防护靠 audit。

INV-2Workspace 是强边界（hard wall）

陈述：跨 workspace 默认完全不可达——user / agent / machine / drive / issue / memory 一律如此。所有高敏感数据（HR / 财务 / 客户隐私 / 未发布产品）通过分配到独立 workspace 隔离，而不是靠 workspace 内的细粒度 file-level ACL。

这是当前阶段最简单、最稳的隐私保护方式。给敏感数据开 hr-workspace / finance-workspace，只对相关员工 grant member——这个 workspace 里的 agent 自动继承隔离，无法 leak 到别处。比维护一套 file-level ACL 矩阵便宜得多，也不易出错。

两层边界，分工明确 边界 1（强）：workspace 是 hard wall，跨墙不可达。  边界 2（轻）：workspace 内部走轻量 ReBAC + 强 transparency——team 表达「谁看哪些 issue/project」，但 team 是组织视图，不是加密边界（见 INV-5）。

§2 · 协作核心

这一节是模型的心脏——agent 怎么被治理、谁对它负责、多人怎么协作。

INV-3Agent 是第一公民，CAN be assignee

陈述：issue 可以直接 assign 给 agent（assignee_type ∈ (member, agent)）。Agent 自驱推进，遇到 block 主动找人。跟 Linear 的 delegate ≠ assignee 取相反立场。

Linear 是 SaaS，agent 是 customer 装的 third-party app，它无法控制 agent 的 accountability 链条，只能保留「human assignee = 追责对象」。Alice 控制整个链条——agent 必有 owner（INV-4），owner 是员工，员工的 accountability 已在 HR/法律上闭环。所以 issue.assignee = agent 在 Alice 合理，在 Linear 不合理。这不是「Alice 更激进」，是「Linear 没有 owner 这个工具」。

INV-4一个 Owner + N 个 Collaborator（agent 和 machine 都适用）

陈述：每个 agent 和每台 machine 都有且只有一个 owner，外加任意多个 collaborator。Owner 权限更高——能改基线、删除、转移、管理 collaborator；collaborator 只有协作权限——能用、能观察、能协同推进，但不能改变这个 agent/machine 的「身份基线」。

Agent — owner vs collaborator

操作OwnerCollaborator
assign task / 调用 / 观察 action stream✅✅
改 system prompt / skill / MCP / 切换 model✅✘
改 permission_mode（user_equivalent ↔ service）✅✘
删除 / archive agent✅✘
转移 owner / 增删 collaborator✅✘

Machine — owner vs collaborator

操作OwnerCollaborator
在上面跑自己的 agent / 查看 machine 状态与 runtime✅✅
改 machine 配置 / 改 visibility（private ↔ public）✅✘
下线 / 删除 / 强制更新 machine✅✘
转移 owner / 增删 collaborator✅✘

「一个责任人 + 多个协作者」让 agent/machine 既能被多人迭代使用，又始终有唯一追责对象。Collaborator 降低了协作摩擦（不必每个人都当 owner），owner 的排他权保证了基线不会被随意改动——改基线这种 intrusive 操作还要再走 INV-9 的 approval。

关键解耦：Owner ≠ Creator ≠ Caller ≠ Machine 拥有者 Owner 是当前的责任主体，跟「谁创建了它」（creator 是历史事实，不变）、「它跑在谁的 machine 上」、「现在谁在调用它」都没有强绑定。Owner 可 transfer（creator 不变）；owner 在 A 部门，agent 可以跑在 B 的 machine 上，被 C 调用。

三个成熟范式都早已把 accountability 从 authorship 解耦：① Apache PMC chair 不一定是 commit 最多的人或 founder，但对项目命运负责，且可传承；② Linux 子系统 maintainer 对走进 mainline 的 patch 负责，patch 作者可能是匿名 contributor——这是 trust delegation 而非 permission delegation；③ GitHub CODEOWNERS 把「对这部分代码负责的人」做成跟 commit history 解耦的 first-class 概念。Alice 是第一个把这套 human-to-human 的解耦应用到 human-to-agent 的产品形态。

INV-5Team 是组织视图，不是加密边界

陈述：Team 在 workspace 内部表达「哪些 member 看哪些 issue / project」。Team 有 public / private 两种 visibility，issue 强制归属于恰好一个 team，project 也归属单一 team。但 team 不是 INV-2 那种 hard wall——它是组织视图，强保护仍靠 workspace 隔离。

可见性规则

  • public team：任何 workspace member 可见其 issue / project。
  • private team：仅 team member 可见。
  • 无 team 的 issue：workspace member 即可见。

Agent 跟 user 一样守 team 边界，不是例外：

  • Agent 属于某 workspace → 不能访问其他 workspace 任何资源（hard wall）。
  • Agent 在 user U 上下文调用 → 看到的 ⊆ U 看到的（user-equivalent，见 INV-8）。
  • U 不在 private team T → U 调用的 agent 也访问不了 T 的 issue / drive。
  • Agent 被直接 assign 到 private team 的 issue → agent 需显式获得该 team 的 access（agent_team_access），与 user 进 team 同理。

INV-6两条正交的权限轴，不要混为一谈

陈述：Alice 的权限由两条互相独立的轴决定。判断「谁能做什么」时必须分开看这两条。

轴取值语义载体
系统角色（subject 在组织里的层级）member / admin / superadminworkspace 级的管理能力；superadmin 是平台级member.role · user.is_superadmin
资源关系（subject 对某个具体资源）owner / collaborator对某个 agent / machine / team 的责任与协作关系owner_id · collaborator 关系表 · team_member.role

两轴正交意味着：一个普通 member 可以是某 agent 的 owner（对它有全权），但对别人的 agent 只是旁观者；一个 admin 有 workspace 管理权，但不会自动变成每个 agent 的 owner。把「管理层级」和「对单个资源的责任」分开，避免了「admin = 所有东西的 owner」这种权力膨胀。

§3 · 权限机制（mechanism over policy）

先把「如何记录/拦截每一次行为」做对（mechanism），再谈「什么行为允许」（policy）。这是 PostgreSQL / Linux / Git 共有的哲学。

INV-7Permission ≠ ACL —— 权限是 intrusive-action 过滤

陈述：不做 fine-grained capability scope / ACL 矩阵。权限模型是两层 intrusive filter：① deterministic rule 拦显著模式（DROP TABLE / push --force / 大批量 export）；② LLM-as-judge 拦语义 intrusive（「这个 query 是不是在绕开 HR tag 看工资表」）。命中 → prompt owner/user 决定。

Capability scope 假设 action space 可枚举；generative agent 的 action space 是 emergent 的。强行 fine-grained 化 = 永远在维护「agent X 能不能调 tool Y」的矩阵 → 维护成本爆炸 → 最后所有 agent 都 grant * → 模型形同虚设。

四个成熟范式都是「拦异常」而非「列白名单」：垃圾邮件过滤（rule + ML，没人列「哪些邮件可收」）· macOS TCC（首次调摄像头才弹窗）· Cursor/Claude Code Plan Mode（每步 bash/write 让你确认）· sudo + audit（不预列能跑什么，而是 require password + 全量记录 + 异常 review）。

INV-8默认 user-equivalent；service mode = admin 门控的例外

陈述：Agent 默认 permission_mode = user_equivalent——task 启动时取 caller 的 ACL 快照，agent 看到的 ⊆ caller 看到的，不可能越权。少数 service mode（agent 能看到 caller 看不到的）必须由 admin 角色创建 + co-sign，并带强 audit + UI 标记 + 季度 review。

Service mode 是 confused-deputy 的合法形态（如 HR bot 需要超越单个 user 的视野），但 Alice 用角色门控而非「任何人都能开」，并让创建路径故意笨拙，防止所有 agent 都升 service mode 绕开限制。

Glean 用 query-time impersonation（agent 只能看到提问者能看到的）实现 user-equivalent；Notion Custom Agent 则把 elevated access 当 feature 公开承认——后者正是我们用角色门控要避免的滑坡。

INV-9Intrusive iteration 需 owner approve

陈述：改变 agent 行为基线的修改（system prompt / 增删 skill / 增删 MCP / 切换 model / 改 permission_mode / 转移 owner）必须 owner approve。非基线修改（名字、头像、tag、UI 颜色、使用统计）和日常 memory 写入（模型自决定）不需要 owner 介入。

把 agent 当成可迭代的「软件 artifact」，用 GitHub PR + CODEOWNERS + branch protection 的同形态 workflow 治理。Owner 只管真正改变行为的变更，避免 approval 沦为 rubber-stamp。

INV-10Audit-over-ACL —— 全量审计，专用工具，与应用观测分离

陈述：所有 agent action 全量 audit，用专用开源 audit 工具（Graylog Open / OpenSearch Dashboards 等，带 UI、append-only、长期保留），跟应用观测（APM / error tracking / metrics）完全分离。当前阶段先把 transparency 做到 100%，再倒推哪些 pattern 需要前置拦截。

Audit 是合规级需求——跟业务 DB 或 Datadog/Sentry 混在一起，audit query 会被业务 noise 淹没，且 ACL 模型不匹配。"After-the-fact reconciliation" 比 "before-the-fact denial" 工程上经济得多。审计后端可替换，对开源 fork 友好。

§4 · 状态模型

INV-11Stateful identity + ephemeral runtime + memory 只做 pointer 层

陈述：Agent 有 stateful identity（DB row + 挂载的 memory）。每个 task 的 runtime instance 由 agent identity ⊕ caller context 合成，task 结束销毁。Memory 能力下放底座（claude / codex / hermes 各自写 jsonl、维护 history），Alice 只做最上层 pointer + metadata 索引，不集中存内容。

三层 state，分清归属

  • Knowledge（不属于 agent）：产品文档、code、issue 历史——在 source-of-truth 系统里，现读现用。当前规模「反 RAG」，不预先 embed。
  • Skills（属于 agent）：操作流程、工具调用模板，cross-agent 可复用。
  • Execution memory（属于 agent，但存在底座）：past task 摘要、owner 标注的 preference。Alice 只存「在哪台 machine / 哪个底座 / 多大 / 何时改」这类 metadata。

底座是可扩展的（已支持 11 种），它们自带 memory 系统——Alice 重写一套只会跟底座 schema 持续打架。Pointer 层让不同 agent 跨实例互访彼此 memory、让 task 结束后有统一地方查询，而实际内容仍由底座原生 API 读写。跨底座 schema 不同时，由调用方用 LLM 概括成 plain text 注入，Alice 不做 schema 转换。

Erlang/OTP：actor 有 PID + state（identity），message handling 是 ephemeral computation。Kubernetes：StatefulSet 表达 stateful identity，Pod 是 ephemeral execution。Alice 的 agent identity / task runtime 是同一分层。

§5 · 关系图速查 + 引用

5.1 关系 tuple（轻量 ReBAC，关系直接建在业务表）

权限 check 退化为这张图上的可达性查询——不引入专门的关系存储引擎（SpiceDB / OpenFGA），因为关系图复杂度低，业务表足够。

关系 tuple语义
workspace:W#member@user:UU 是 W 的成员（hard wall 的准入）
team:T#member@user:UU 是 team T 成员（workspace 内细分可见性）
team:T#member@agent:AA 获得 team T 的 access（assign 到 private team 时需要）
issue:I#in_team@team:Tissue I 属于 team T（恰好一个）
issue:I#assignee@agent:AA 在做 I（第一公民 assignee）
agent:A#owner@user:UU 对 A 负责（唯一）
agent:A#collaborator@user:UU 协作使用 A（多个）
machine:M#owner@user:UU 拥有 machine M（唯一）
machine:M#collaborator@user:UU 可在 M 上跑 agent（多个）

5.2 一条典型可见性判断

CanViewIssue(subject, issue):          // subject 是 user 或 agent
    1. same_workspace(subject, issue)  ?  继续 : 拒绝     // INV-2 hard wall
    2. issue.team_id is null           ?  允许（workspace member 可见）
    3. team.visibility == 'public'     ?  允许
    4. is_team_member(subject, team)                      // INV-5
