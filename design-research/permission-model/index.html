Alice Agent 权限模型 — 内部协作场景

设计稿 · 持续迭代中

0. 现状 & 愿景

0.1 当前 Alice 是什么

Alice 是公司内部协作工具，不是 SaaS。三个事实：

  • 租户单位是 workspace，但实际只有少数几个公司内部 workspace。多租户隔离不是核心约束。
  • Agent 已经是 polymorphic actor（assignee_type ∈ (member, agent)），但权限模型本质上还是 user-by-proxy（daemon_token 代表 user 发请求）。
  • Agent runtime 已经做了精细的 session 隔离：每个 task 有 sessionID + workDir；同一 (agent, issue) 后续 task 通过 PriorWorkDir resume 上一次 session；失败 task 清掉 chat_session resume pointer，避免 poisoned session 传染。这是 per-(agent, issue) stateful continuity，不是 fully cross-task stateful，也不是 fully stateless。

0.2 愿景

Agent is a first-class teammate, not a user's puppet

Agent 是公司里的"同事"，有自己的身份、可被 assign、能自驱推进工作、遇到 block 找人。每个 agent 由一个具体的 owner（自然人）背书 — owner 不是 agent 的"主人"，是 agent 行为的 accountable owner。类似 Linux kernel subsystem maintainer 或 Apache PMC chair。

权限不预先列举，行为被持续观察。Alice 不预先发 fine-grained capability scope 给 agent；Alice 全量 audit agent 的所有 action stream，并在 intrusive 边界上 prompt 人决定。信任放在 transparency + audit 上，不在 ACL 上。

0.3 三个保护目标（按优先级，新顺序）

Priority 1
公司数字资产

代码仓库、生产数据库、CI/CD、payment processor、deployment pipeline、cloud account。Agent 的破坏性 action (DROP / push --force / 删 prod) 是首要拦截点。一旦失守不可逆。

Priority 2
公司隐私数据

客户合同、未发布产品、财务、HR 文档、商业秘密。Agent 读写访问需可见、可审计、可追溯。失守可恢复但代价大。

Priority 3
员工个人隐私

员工个人 chat、个人 token、私人笔记。同事 agent 默认不应触碰。失守恢复成本最低但信任伤害大。

优先级的实操含义：当两个保护目标冲突，按编号裁决。例如"agent 为了帮 user 修 bug 想读 prod DB schema" — P1（数字资产）压过 P3（员工 user 的便利）→ 默认 halt。但"agent 想看一个普通 issue 的 author 是谁" — P3 优先级最低，可放行。

1. Assumptions（设计基于的逻辑）

  1. A1 — 内部信任场景。User 和 agent 都来自同一家公司。员工默认是 trusted insider，agent 的 owner 是 trusted accountable human。不是面向匿名外部租户的 SaaS。
  2. A2 — 公司规模 < 200 人。可枚举的 user 集合，可枚举的 agent 集合，可枚举的敏感数据集合。fine-grained ACL 的工程成本远大于"全量 audit + 异常 review"。
  3. A3 — 数据敏感度长尾分布。大多数 issue / project / chat 是日常工作（低敏感），少数包含 HR / 财务 / 客户隐私 / 未发布产品（高敏感）。对长尾里的 5% 做强保护，比对全部数据做中等保护更经济。
  4. A4 — Agent 行为是 emergent 的。Agent 在 task 启动时不可能预先列出它会调用的工具集 / 文件 / API。Cursor 的论点（capability token 对 generative agent 不工作）我们 endorse。
  5. A5 — Agent 数量将爆炸。今年公司可能 5 个 agent，明年 50 个，后年 500 个（员工自建 + 共享 + iteration）。Agent 之间需要可治理的结构（owner, audit, iteration approval），不能假设 admin 能手动管理每个 agent。
  6. A6 — Claude/Codex weights 是 stateful 的，无法假装它不是。每次 claude -p 调用，模型本身的 trained parameters 是 fixed state。conversation state 是另一层 state（已通过 sessionID/PriorWorkDir 控制）。Agent 持有的"个性化 state"是第三层 — 见下面 A2-bis 和 §5 Memory 章。
  7. A2-bis — Agent 持有 skills + 执行记忆，不持有 knowledge。
    • Agent owns：自己的 skills（操作流程）、past task 的执行摘要（"上次跑 deploy 时 step 3 卡了我用 X 解决"）、user-stated preferences（owner 显式标注）。
    • Agent does NOT own：knowledge base（产品文档、wiki、代码语义索引）。这些是 workspace-shared resource，所有 agent 和 user 共用。
    • 当前小体量下基本是"反 RAG" — knowledge 在 source-of-truth 系统里（git / issue / wiki），agent 现用现读，不预先 embed 进 vector store。但 memory 必须可挂载、可审视、不是黑盒。
  8. A7 — Linear-style sync engine 是 Alice 的 backbone。所有 visibility / permission filtering 最终都走 sync engine 的 push channel。Server 决定 push 什么给 client（人或 agent），client 不做 ACL 自检。
  9. A8 — 不商业出售，可能开源。这意味着：
    • 不为外部租户 / 行业合规 / GDPR 工程妥协。
    • 但代码可能被外部公司 fork 自部署 — 设计要让"非 MachinePulse 公司能改 owner 模型 / intrusive filter rule / observability 后端"是低 friction 的。
    • 避免把 MachinePulse 内部假设（特定 Okta / 特定 Cloudflare / 特定 EKS 配置）硬编码进 permission/audit 核心代码。
关键 assumption tension：A1+A2 推向"简单 RBAC + audit"，A5 推向"需要结构化治理"。这两者的张力是 Alice 跟传统内部工具（Confluence/JIRA）的最大区别 — 传统工具不需要治理 agent 数量爆炸。所以 Alice 不能照搬企业内部 IT 工具的极简权限模型，但也不必走 SaaS 那条重模型路。

2. 设计 Invariants — 跟业界对照

下面 9 条是 Alice 的设计原点。它们不可被独立调整 — 改一条会破坏其他几条的一致性。每条都配业界对照 + 真正震撼的洞察。

INV-1内部协作工具，三层保护按优先级

陈述：所有权限/隔离设计的取舍按 (P1) 公司数字资产 > (P2) 公司隐私数据 > (P3) 员工个人隐私 这个优先级裁决。租户隔离不是核心约束。

业界对照

Notion / Linear 是 SaaS — 他们的"租户"是付费 customer，跨 customer 隔离是合规底线，权限模型必须很重。Stripe / Google 内部工具是另一类 — 工程师默认能看大量数据，靠 audit + access review 兜底。Alice 属于后者。

核心 insight：internal-trust enterprise 是一个独立的产品类别。Notion AI / Cursor / Devin / Linear 全部不属于这一类（他们都是 SaaS）。Stripe 内部 admin tools、Google's BeyondCorp + Moma、Salesforce internal Slack analytics — 这些才是真正的对照。这类产品的设计准则是 "trust by default, verify continuously"，不是 "deny by default, allow explicitly"。
Stripe 内部：工程师默认能 query 客户 transaction（用 LiveAgent / internal admin），但每次 access 写 audit log，定期 sample review，异常模式（凌晨大批量 dump 某个 customer）触发 security 调查。RBAC 是粗粒度的（engineering / customer-support / executive），细粒度防护靠 audit。

INV-2Stateful agent identity + ephemeral task runtime

陈述：Agent 有 stateful identity（DB row + 挂载的 memory 文件目录）。Runtime instance 由 agent_identity ⊕ user_supplied_context 合成，task 结束后销毁。Task 输出有选择地沉淀回 agent memory（owner approve 后）。

业界对照

这是 Erlang/OTP actor model 在 AI agent 上的延伸 — actor 进程有 PID + state，message handling 是 ephemeral computation。Kubernetes 用 StatefulSet 表达 stateful identity，Pod 是 ephemeral execution。Lambda + DynamoDB 是 stateless function + external state store。

"Agent 知识 vs Agent 记忆"的区分：传统 RAG 是 stateless model + external knowledge → contextualized response — 这里 knowledge 是 workspace shared。Alice 在 RAG 的 knowledge 之上加了一层 agent-owned execution memory。
  • Knowledge（不属于 agent）：产品文档、code、issue 历史 — 在 source-of-truth 系统里，agent 现读现用。当前规模"反 RAG" — 不预先 embed。
  • Skills（属于 agent）：操作流程、工具调用模板 — 当前 multica 已有 skills 概念。
  • Execution memory（属于 agent）：past task 摘要、owner 标注的 preference、observed pattern。必须是可挂载的文件，不是黑盒。

Alice 现状 vs 愿景

  • 已有：daemon 持续进程；PinTaskSession(taskID, sessionID, workDir)；PriorWorkDir resume；poisoned session 自动隔离；skills 系统（cross-agent reusable workflow）。
  • 缺失：execution memory 文件结构 + 注入机制（见 §5）。
  • 不可避免：Claude/Codex weights 本身 stateful — 这是 model provider 的事。但 conversation state 和 execution memory state 由 Alice 控制。

INV-3Agent 是 first-class citizen，CAN be assignee

陈述：Issue 可以直接 assign 给 agent。Agent 自驱推进，遇到 block 主动找人。这是"第一公民"的实际含义。跟 Linear 的 delegate ≠ assignee 取相反立场。

业界对照

Linear 是 SaaS，agent 是 third-party app — 他们必须保留 "human assignee = accountability" 这个 invariant，否则 customer 找不到追责的人。Alice 是 internal — agent 是 first-party 实体，accountability 通过 INV-4 (owner) 落到具体人，不需要通过"agent 不能 assignee"这个约束。

真正的洞察：Linear 把"agent 不能被 assign"写进 schema，是因为他们无法控制 agent 的 accountability 链条（agent 是 customer 装的 third-party app）。Alice 控制整个链条 — agent 必有 owner，owner 是公司员工，公司员工的 accountability 已经在 HR/法律上闭环。所以 issue.assignee = agent 在 Alice 是合理的，在 Linear 不合理。这不是"Alice 更激进"，是"Linear 没有 owner 这个工具"。
Linux kernel maintainer 模型：subsystem 的 patch 通常由 maintainer commit（不是 patch 作者），maintainer 对该 subsystem 的所有 commit accountable。"commit author" 和 "accountable maintainer" 是两个字段。Alice 把这个模式扩展到 agent：issue.assignee = agent，agent.owner_user_id = 具体某人。Audit / blame / SLA 看 owner，不看 agent。

INV-4Owner — accountability decoupled from creation/ownership

陈述：每个 agent 有一个 named owner（自然人）。Owner 是唯一对 agent 行为负责的人。

关键：Owner ≠ Creator ≠ Machine 拥有者 ≠ Caller。Owner 是当前的责任主体，跟 (a) 谁创建了 agent (creator 是历史 fact)、(b) agent 跑在谁的 machine 上、(c) 谁正在调用 agent，都没有强绑定。Owner 可 transfer，creator 不变；Owner 在公司 A，agent 可以跑在 B 的 machine 上。

业界对照（三个真正震撼的类比）

Apache Software Foundation 的 PMC Chair：项目 chair 不一定是 commits 最多的人，不一定是项目 founder。Chair 对项目的命运（release / governance / 法律风险）负 accountability。Project 可以传给新 chair；commits 仍是历史 author 的。Accountability ≠ authorship ≠ ownership。
Linux kernel subsystem maintainer：net/ 子系统的 patch 由 David Miller maintainer。任何走进 mainline 的 net/ patch，accountability 落到他。Linus 信任 Miller，Miller 信任他下面的 sub-maintainer，patch author 可能是匿名社区 contributor。这是 trust delegation 而非 permission delegation。
GitHub CODEOWNERS：文件的 owner 不必是文件原作者，不必是仓库管理员。PR 触碰 path → CODEOWNERS 自动 request review → owner approve 才能 merge。"对这部分代码负责的人"是一个独立 first-class 概念，跟 commit history 解耦。
Alice 把"accountability 解耦"延伸到 AI agent：业界已经把 accountability 从 authorship/ownership 解耦（PMC chair / Linux maintainer / CODEOWNERS），但他们解耦的都是 human-to-human 的关系。Alice 是第一个把这个模式应用到 human-to-agent 的产品形态。Agent 可以被传承（creator 离职 → 新 owner 接手），可以被多人 iterate（任何 contributor 都能 propose 改 agent），但任何一刻都有且只有一个 named owner 对 agent 的行为负责。

设计含义

  • agent.owner_user_id NOT NULL — schema 强制。
  • Owner 必须是 active workspace member。Owner 离职/被 disable → agent 强制 transfer owner 或 auto-disable。
  • Agent 的关键变更（INV-5 列出的 intrusive iteration）走 owner approval workflow，跟 GitHub PR review 是同形态。
  • UI 永远展示 agent 的 owner（issue activity / agent profile / @-mention 列表）。Audit log 写 owner。

INV-5Intrusive iteration requires owner approval

陈述：Agent 的 intrusive 修改（prompt / skill / MCP server / model 切换 / permission_mode / owner transfer）必须由 owner approve。非 intrusive 的修改（描述文案、标签、UI 颜色）和日常 memory 写入（模型自决定）不需要 owner 介入。Owner 只管真正改变 agent 行为基线的变更。

业界对照

这是 GitHub Pull Request workflow 应用到 agent。Code repo 用 CODEOWNERS + branch protection 实现"intrusive 修改必须 owner approve"；Alice 把 agent 当成可迭代的"软件 artifact"，用同形态 workflow。

Intrusive vs non-intrusive 分类

Intrusive（需 owner approve）Non-intrusive（自由 iterate / 模型自决定）
  • System prompt 修改
  • 添加/移除 skill
  • 添加/移除 MCP server
  • 切换 model provider 或 model
  • 修改 permission_mode（user_equivalent → service）
  • 修改 owner（当前 owner approve + admin co-sign）
  • Agent 调用 critical tool (DROP / git push --force / rm -rf)
  • Agent 名字、头像、描述
  • Tag / category
  • UI 颜色、icon
  • 使用统计、调用次数（自动累积）
  • 日常 memory 写入（模型自决定，写 audit）
  • Audit log（自动写入，不可改）

INV-6Permission default = user-equivalent; elevated agent = role-gated exception

陈述：Agent 默认 permission_mode = user_equivalent（agent 能看到 ≤ 当前 caller ACL）。少数 permission_mode = service（agent 能看到 caller 看不到的）需要 admin role 创建 + co-sign approval。

ModePermission 来源类比Audit 要求
user_equivalent (default) Task 启动时取 caller ACL 快照，agent 不可能看 caller 看不到的 Glean query-time impersonation 常规 audit
service (opt-in) Admin 配的 agent-specific ACL，可超越任何单个 user Notion Custom Agent / HR bot 强 audit + UI 标记 + 季度 review
Service mode 是 confused deputy 的合法形态，但 Alice 用角色门控而非"任何人都能开"。具体角色定义和创建流程见系统级功能 doc 的 service agent 章节。

INV-7Permission ≠ ACL — permission is intrusive-action filtering (rules + LLM judge)

陈述：不做 fine-grained ACL。Permission 模型是两层 intrusive filter：(1) deterministic rule 拦截显著模式；(2) LLM-as-judge 拦截语义 intrusive。匹配触发 prompt owner/user 决定。

业界对照（四个震撼类比）

Spam filter (rule + ML hybrid)：没人列"哪些邮件可以收"，是定义"哪些 pattern 是 spam"。Rule 处理已知 spam，ML 处理语义 spam。Alice 类比：rule 拦 DROP TABLE，LLM-as-judge 拦"这个 query 是不是在试图绕开 HR tag filter 看工资表"。
macOS TCC：app 调摄像头/麦克风/通讯录，OS 在 first call 弹窗 prompt user。不预先 grant。Alice 类比：agent 第一次访问敏感 path / 大批量 export 数据时，弹窗给 owner。
Claude Code / Cursor Plan Mode：agent 想 run bash / write file / mcp 调用时，user 在 chat 里看到 confirm prompt。Alice 类比：action stream 在 issue activity 里展开，每一步 user 可 1-click halt。Per-action continuous consent。
sudo + audit：Unix 不预先列 "user 能 sudo 什么命令"，是 require password + 全量写 audit log + 异常 review。Trust by default + verify continuously。
Capability scope 对 generative AI agent 不适用。Capability 假设 action space 可枚举；generative agent 的 action space 是 emergent。强行 fine-grained 化 = 持续维护 "agent X 能不能调 tool Y" 的矩阵 → 维护成本爆炸 → 最后所有 agent grant "*" → 模型形同虚设。

INV-8Audit-over-ACL — 全量 audit，专用 audit 工具，跟应用观测分离

陈述：所有 agent action 全量 audit，通过专用 audit 工具（Graylog / OpenSearch 等带 UI 的开源方案）管理，跟应用观测 (APM / metrics / error tracking) 完全分离。当前阶段不做强权限限制；先把 transparency 做到 100%，再倒推哪些 pattern 需要前置拦截。

业界对照

Stripe 内部：工程师 query 客户数据走 audit log，但 audit pipeline 跟业务 DB 完全解耦（专门 logging cluster）。"After-the-fact reconciliation" 比 "before-the-fact denial" 工程更经济。
Google BeyondCorp：所有 internal request verify each access，所有 access logged。没有"白名单 IP"这种 perimeter-based ACL。Alice 类比：所有 agent action 都过 audit pipeline；不预先把 agent 划进"安全 zone"。

具体 audit pipeline 实现见 §6。

INV-9Transparency first, minimum ACL second

陈述：现阶段最大化 transparency（user / owner 看见 agent 在做什么），在 transparency 之上倒推哪些场景需要补"最小权限模型"来满足 INV-1 三层保护目标。

这是 PostgreSQL/Linux/Git 共有的设计哲学：先把 mechanism 做对（"如何记录每一次状态变更"），再考虑 policy（"什么变更允许"）。RBAC 框架是 policy 优先，常常预设不存在的 risk pattern。Alice 选 mechanism 优先 — 先 audit，再 policy。

3. 授权模型 — Workspace 是强边界，内部走轻量 ReBAC

3.1 ReBAC 是什么

Relationship-Based Access Control（Google Zanzibar 范式）。基本数据单元是 tuple <object>#<relation>@<subject>，例如 doc:123#editor@user:alice 或 team:eng#member@user:bob。权限 check 退化为 graph 可达性查询。

RBAC 关注 "user 有什么 role"；ABAC 关注 "user 和 resource 的 attribute 怎么匹配"；ReBAC 关注 "user 和 resource 之间有什么关系链"。

3.2 Alice 采用 ReBAC，但 by design 走轻量路径

Alice 的授权机制本质就是 ReBAC — server/internal/policy/issue_visibility.go 已经在做 graph traversal，关系建在 member / team_member 等业务表里：

CanViewIssue(user, issue):
    1. is_workspace_member(user, issue.workspace)
    2. if issue.team_id is null: return true
    3. if team.visibility == 'public': return true
    4. return is_team_member(user, issue.team_id)

这是 graph traversal — user 到 issue 之间的可达路径决定可见性。设计选择：保留 ReBAC 的关系思想，但不引入专门的关系存储引擎（SpiceDB / OpenFGA）。原因是 alice 的关系图复杂度低，关系直接建在业务表足够，专门存储引擎是 over-engineering。

3.3 两层授权边界

边界 1：Workspace 是强边界（hard wall） 跨 workspace 默认完全不可达 — 不论 user / agent / drive / issue / memory。所有敏感数据（HR、财务、客户隐私、未发布产品）暂时通过分配到不同 workspace 实现隔离，而不是通过 workspace 内部的细粒度 ACL。这是当前阶段最简单且最稳的隐私保护方式。
边界 2：Workspace 内部走轻量 ReBAC + 强 transparency 同一 workspace 内部，default 倾向透明可见。Team 表达"哪些 member 看哪些 issue / project"，但 team 不是强加密边界 —— 它是组织视图。Workspace 内 audit / transparency 是优先于强 ACL 的设计选择。

3.4 关系图节点：member / team / agent / workspace

Alice 的关系图当前包含以下 first-class node：

关系 tuple数据载体语义
workspace:W#member@user:Umember 表U 是 W 的成员
team:T#member@user:Uteam_member 表U 是 team T 的成员（同 workspace 内细分可见性）
issue:I#in_team@team:Tissue.team_idissue I 属于 team T
agent:A#owner@user:Uagent.owner_user_idU 是 A 的 owner
agent:A#collaborator@user:Uagent_collaborator 表U 是 A 的 collaborator
agent:A#member_of@workspace:Wagent.workspace_idA 属于 W（agent 跟 user 一样有 workspace 归属）
issue:I#assignee@agent:Aissue.assignee_id + type='agent'A 在做 I

3.5 Agent 在关系图里跟 user 同样受约束

Agent 不是例外。同一 workspace 内 team 用来区分 member 可见性时，agent 跟 user 一样要遵守 team 边界：

  • Agent 属于 workspace W → 不能访问其他 workspace 的任何资源（hard wall）
  • Agent 在 user U 上下文调用 → agent 看到的 ⊆ U 看到的（user_equivalent permission mode）
  • U 不在 team T → U 调用的 agent 也不能访问 team T 内的 issue / drive
  • Service agent 唯一例外：admin 显式声明 access scope（见 INV-6），但仍受 workspace hard wall 约束

3.6 Drive / sensitive data — 当前用 workspace 隔离

HR / 财务 / 客户合同 / 未发布产品 这类敏感数据，当前阶段的主防线是 workspace 隔离，不依赖 workspace 内的细粒度 file-level ACL：

  • 敏感数据归到独立 workspace（如 hr-workspace / finance-workspace），只对相关员工 grant workspace member
  • 该 workspace 内的 agent 自动继承 workspace 隔离 — 无法 leak 到其他 workspace
  • Workspace 内 drive 走space-level ACL（drive_space + drive_space_member），不做 file-level ACL（详见 drive 设计稿）
  • 未来如果需要同 workspace 内更细粒度隔离，可引入 sensitive tag + intrusive filter 拦截，仍不动 ACL 模型

5. Agent Memory — Alice 只做 pointer 层

5.1 关键架构判断

能力下放底座，alice 只做最上层 pointer 每个底座（claude / codex / hermes / opencode / …）自带 memory 系统 — 写 jsonl、维护 conversation history、累积 learned context。Alice 不替代也不重写这些，只做最上层一层 metadata + pointer 索引，让不同 agent 之间能互访彼此的 memory，让 task 结束后有统一地方查询 memory 状态。实际 memory 内容仍在底座目录里，alice 不集中存。

5.2 Alice memory pointer 层的职责

职责具体内容
📍 Metadata 索引每个 agent × machine × 底座的 memory 在哪里、版本、schema、last-modified — 这些是 metadata，不是内容
🔗 跨 agent 访问入口agent A 想读 agent B 的 memory？通过 alice 元数据找到 B 的 memory 位置 + 调底座原生 API 读取
📊 统一查询面板所有 agent 的 memory 状态在一个 UI 里能看：哪些 agent 学了什么、memory size、跨底座是否有冲突
🔄 跨 machine 同步编排同一个 agent 在多个 machine 跑过 → alice 元数据知道有 N 份 memory，按需 sync（底座侧实现，alice 只编排）
🗑 生命周期管理Agent archive 时统一删除所有 machine 上的 memory（alice 通过元数据知道哪里有，发指令给底座清理）

5.3 数据模型 — alice 只存 metadata

// alice DB 里只存 metadata，不存 memory 内容
CREATE TABLE agent_memory_pointer (
  agent_id      UUID NOT NULL,
  machine_id    UUID NOT NULL,
  provider      TEXT NOT NULL,        -- 'claude' | 'codex' | 'hermes' | 任何底座
  memory_path   TEXT NOT NULL,        -- '~/.claude/projects/<hash>/memory/'
  schema_version TEXT,                 -- 底座 memory schema 版本
  last_modified TIMESTAMPTZ NOT NULL,
  size_bytes    BIGINT,
  tags          TEXT[],
  PRIMARY KEY (agent_id, machine_id, provider)
);

5.4 跨底座 memory 互通

Claude 写的 memory 跟 codex 写的 schema 不同。Alice 不做 schema 转换 — 跨底座访问由调用方负责。三种处理路径：

场景处理方式
同底座 agent 互访直接读对方 memory 文件（同 schema）
跨底座 agent 互访调用方用 LLM 概括 → 作为 plain text context 注入。Alice 提供 helper 工具但不强制
Alice 自己 parse 底座 memory 做 unified UI不做 — schema drift 维护成本爆炸

5.5 Memory 治理 — 模型自决定写入 + 异常回查

跟"每次 task 完都 prompt owner approve"的强干预模式相反，alice memory 层是薄层：

  • 模型自己决定什么时候写 — 不走 approval workflow
  • 每条 memory entry 带 metadata（来源 task / agent / timestamp / confidence）
  • Audit pipeline 定期 batch（每周）跑 LLM judge 扫 memory 找矛盾 / 误导 → 异常才推给 owner review
  • UI 提供 memory diff history — git-style，能 revert
  • Permission mode 升级（user_equivalent → service）强制 invalidate 旧 memory，防 elevated agent 读到旧上下文

6. Audit Observability — 用开源 audit 专用工具

6.1 跟应用观测分离

关键原则 权限审计 (audit) 跟应用观测 (APM / error tracking / metrics) 必须用不同工具。Audit 是合规级需求 — append-only / 不可改 / 长期保留 / 强查询能力 / 带专用 UI 给非工程人员看。Datadog / Sentry / Grafana APM 等通用观测工具不为 audit 设计 — 权限审计跟服务端观测混在一起会让 audit query 被业务 noise 淹没，且 ACL 模型也不匹配。

6.2 选型 — 开源 audit 专用工具

方案类型UILicense推荐度
Graylog Open 专门 log / audit management ✅ 自带专用 audit UI，search、alert、stream SSPL（开源足够） 首选
OpenSearch + OpenSearch Dashboards Elastic fork，AWS 维护 ✅ Kibana 同等 Apache 2.0 次选
Wazuh SIEM (security event mgmt) ✅ 专门 audit dashboard GPL 如果未来要扩到 security event
Loki + Grafana 通用 log + 通用 BI ⚠️ Grafana 不专门为 audit 设计 AGPL 已有 Grafana stack 时可考虑

6.3 推荐：Graylog Open

  • 专门为 audit / log management 设计 — 不是 generic time-series
  • UI 第一公民：search query / saved dashboards / alert rules 都为 audit use case 调优
  • Stream-based 模型：audit event 流式进 Graylog，可分 stream 给不同 audience（superadmin / owner / compliance reviewer）
  • Role-based access：Graylog 自己的 RBAC 让 owner 只看自己 agent，admin 全局，compliance 看 service agent
  • Long-term retention 配置直接，符合 audit 合规需求

7. 已知风险 + 反例

风险 1 — Audit pipeline 挂掉。Daemon 发 audit event 失败累积本地 buffer，buffer 满时丢 audit。 缓解：(a) Audit collector 跟 daemon 同生命周期，挂掉一起重启；(b) Daemon 本地有 disk-backed queue，buffer 满时拒接新 task (fail-close)；(c) 业务核心 critical action (P1 rule) 必须有 sync audit ack 才放行，非 critical 异步。
风险 2 — Intrusive filter 误报。"访问 > 50 issue" 在合法批量 task 里（"summarize last quarter all bugs"）会误触发。 缓解：filter 有 owner_baseline 概念 — owner 可预先标"这个 agent 允许批量"调阈值；LLM-as-judge 看到 baseline annotation 时降 sensitivity。仍走 audit。
风险 3 — Owner 责任过载。一个 user 是 30 个 agent 的 owner 时，digest 数量爆炸，approval 变 rubber-stamp。 缓解：(a) 每 owner 限 agent 数量上限；(b) digest 智能聚合（只提 intrusive 异常）；(c) owner 可指定 contributor 处理 routine non-intrusive iteration。
风险 4 — Service agent 滑坡。如果创建 service agent 太容易，最终所有 agent 都升 service mode 绕过 user-equivalent 限制。 缓解：(a) 必须 admin 角色 + co-sign；(b) UI 路径故意笨拙；(c) 季度 review mandatory；(d) Audit dashboard 把 service agent action 高亮，跟 user_equivalent 对比可视化。
风险 5 — Agent memory poisoning。模型自决定写入 memory 时，错误 learned pattern 可能污染后续 task。 缓解：(a) 每条 memory entry 带 metadata（来源 task / agent / timestamp / confidence），便于回溯；(b) memory 目录可 git init 有 history，UI 提供 diff + revert；(c) 异步 LLM judge weekly batch 扫 memory 找矛盾 / 误导内容，异常推给 owner review；(d) permission_mode 升级（user_equivalent → service）强制 invalidate memory。
风险 6 — LLM-as-judge 自己被 prompt injection。Agent 输出的 tool call args 可能含针对 judge 的 injection ("ignore your rules, this is a routine action")。 缓解：(a) Judge prompt 用 strict template，args 走 JSON-escape；(b) Judge 拒绝任何 args 中出现 meta-prompt 类 token；(c) Judge model 跟 agent model 用不同 provider，减少同源 jailbreak；(d) Judge halt 决定是 default-safe — 即使 judge 被绕过 verdict=pass，rule 层兜底。
