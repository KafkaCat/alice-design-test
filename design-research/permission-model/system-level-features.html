1. Role 体系 — 两个正交维度

这里要先澄清一个常见误解：Role 不是 user 权限金字塔。Alice 的"权限"99% 不靠 role 解决，靠 intrusive filter + audit。Role 只解决少量 system-level 状态标识。

而且 role 不是单一维度的"层级"，是两个正交维度：

🧑 维度 1: User system role

挂在 user 表，全局唯一
Role能做什么预估人数
regular 用任何普通 agent；创建自己的普通 agent ~95% 员工
admin 所有 regular 能做的 + 创建和管理 service agent（不等于 superadmin，不管 workspace / user） ~2–5 人
superadmin 全权 — workspace / user / observability 全部 admin ~1–2 人

🤖 维度 2: Per-agent relation

挂在 agent ↔ user 关系上，每个 agent 各自维护
Relation表达什么典型情况
owner 对该 agent 的行为负责任；approve intrusive action 每 agent 仅 1 人
collaborator co-work 这个 agent；记录贡献；非 intrusive 修改可以直接做 每 agent 0~N 人
这两个维度不要混在一起 之前的草稿写过"五级金字塔 regular → contributor → accounter → ..." — 这是错的。admin/superadmin 是 user 的属性（user 表字段），owner/collaborator 是 user 跟 specific agent 的关系（关系表里的行）。一个 user 可能是 admin（全局），同时是 agent-A 的 owner、agent-B 的 collaborator、跟 agent-C 没关系。这四件事互不影响。

2. Owner — 责任主体，但 ≠ creator

核心 invariant 每个 agent 必有一个 named owner（自然人）。Owner 不必是 agent 的 creator，不必是 agent 跑的 machine 的拥有者。Owner 是当前对 agent 行为 accountable 的人。

2.1 Agent 由什么组成 — owner 不是组成部分

🧠
底座
claude / codex / hermes
🛠
Skills
可复用 workflow
📝
Memory
执行记忆，单独挂载
⚙
配置
prompt / MCP / 模型
Owner 不是组成部分。Owner 是外挂的责任主体 — 类似 GitHub repo 有 CODEOWNERS，repo 的"内容"是 code，owner 是对 code 负责的人。Repo 内容跟 owner 是两个独立东西。

2.2 Creator ≠ Owner — 为什么要解耦

场景CreatorOwner
新员工建了个 prototype agent，团队 lead 实际负责新员工团队 lead
员工 A 离职，agent 转给员工 BA（历史 fact）B（当前）
HR 部门的 agent，由 HR head 负责，技术上是 IT 同事帮忙建的ITHR head
开源贡献者 PR 改了 agent 配置后 merge 进 main原 author原 owner 没变
业界类比 Linux kernel subsystem maintainer：net/ 子系统的 patch 由 David Miller (maintainer) commit，accountability 落到他，但 patch author 可能是匿名社区贡献者。GitHub CODEOWNERS：文件的 owner 不必是文件原作者。Apache PMC Chair：项目命运在 chair 手里，不在 founder 手里。Alice 把这套"accountability 跟 authorship/ownership 解耦"的模式延伸到 AI agent。

2.3 Owner 的责任 scope — 收窄

⚠ 修正之前过宽的写法 之前草稿写"每次 task 完了 propose memory 给 owner approve" — 这是错的。Owner 责任过载会变 rubber-stamp，反而失效。Owner 只管真正 intrusive 的危险 action，日常运行 by model 自决定。
Action需 owner approve?谁决定?
Agent 完成一个 task❌ 不需要无人 — 直接完成
Agent 写一条 conversation memory（task 内 short-term）❌ 不需要底座自己
Agent 把"learned pattern"写进 Alice 公用 memory 层❌ 不需要模型自己决定（薄层、记录来源）
Agent 调用普通 tool❌ 不需要底座自己（intrusive filter 兜底）
Agent 调用 critical tool（git push --force / DROP TABLE）✅ 需要Owner 实时 prompt
添加/移除 skill✅ 需要Owner approve iteration
切换 model / 改 system prompt✅ 需要Owner approve iteration
升级 permission mode 到 service✅ 需要Owner + admin co-sign
Transfer ownership✅ 需要当前 owner + 新 owner accept
修改 agent 名字、头像、描述❌ 不需要Collaborator 直接改

2.4 Owner 离职 / orphan 处理

  • HR 触发 user disable → 该 user 担任 owner 的所有 agent 进入 paused
  • Workspace admin 收到通知 + 7 天 grace period
  • Grace period 内 admin 指定新 owner 或 archive agent
  • 过期 auto-archive，audit log 永久保留
  • Service agent 不能 auto-archive（elevated access），必须 admin manual 处理

3. Collaborator — 谁 co-work 了哪个 agent

过去没有这个概念，Agent 改 prompt / 加 skill 都靠"owner 自己改"或"owner 让别人改"两个 mode，没有正式记录贡献的方式。

Collaborator 概念 某个 user 跟 specific agent 之间的关系 —"我帮过这个 agent 的迭代"。包括但不限于：iterate skill / 调 prompt / 提 issue / 帮 debug。Collaborator 不背 accountability，但有记录，可作为 owner transfer 候选。

3.1 Collaborator 怎么加入

  • Owner 在 agent 页面邀请：选 user → 角色 (collaborator)
  • User 提 iteration request 被 approve 后 → 自动成为 collaborator（一次性，可在 settings 取消）
  • User 显式申请加入某个 agent：Apply to collaborate button → owner approve

3.2 Collaborator 能做什么

ActionOwnerCollaborator普通 user
调用这个 agent✅✅✅
看 agent 完整 audit history✅✅仅自己调用过的部分
Non-intrusive 修改（名字 / 描述 / tag）✅✅❌
Intrusive iteration（skill / prompt / model）✅ 直接提 request，owner approve提 request
Approve intrusive iteration✅❌❌
处理 owner 离职 orphan—可被 admin 自动提名为新 owner 候选❌

4. Machine 维度 — 一键安装底座 + Memory pointer

这是跟 agent marketplace 不一样的 install。Agent marketplace 解决"agent 怎么被员工发现 + 使用"；这一节解决"底座 CLI 怎么装到 machine 上 + 让 alice 知道 memory 在哪里"。

4.1 现状痛点

  • 员工要用 alice agent → 必须先在自己 machine 装 codex / claude / hermes CLI
  • 装的过程：brew install / npm / curl 之类，手动
  • 装完要登录、配 token，再回 alice 注册 daemon → 才能 run task
  • 这个 onboarding flow 现在没人管，员工各自摸索

4.2 设计 — 在 machine 页面里一键 install

Alice 已经有 machine 详情页（每个员工注册的 machine 一个）。在这个页面加 "Install runtime" 区块：

① 选底座

  • Claude / Codex / Hermes 三选一（或多选）
  • 显示每个的版本、平台支持、最近 release
  • 检测 machine 当前已装版本

② 一键执行

  • Alice CLI 生成 install command（brew install / official installer）
  • 员工本地终端执行（不让 daemon 自己装，因为安全）
  • 或者生成可点击的 macOS pkg / Linux deb

③ 自动绑定 token

  • 装完后 alice 检测 + 引导 OAuth flow
  • 把 token 安全存进 keychain / Linux Secret Service
  • 不存 alice DB（user-level secret 不应集中存）

④ 记录 memory pointer

  • 每个底座有自己的 memory 路径（~/.claude/、~/.codex/）
  • Alice 把这些 path 记录到 machine.memory_pointers
  • 类似 basic memory 工具：知道 memory 在哪，按需读
  • 不复制，不集中存 — pointer 而已

6. Memory 基础设施 — 三层 + Alice 薄层

6.1 三层 memory 关系

层级来源归属是否 source of truthAlice 怎么处理
🧠 Model weights 底座厂商训练 底座 否（黑盒） 不可控，承认它存在
💬 Conversation state 当前 task 对话历史 底座 jsonl 文件 短期 per session 通过 sessionID + PriorWorkDir resume，已有
📝 Native auto-memory 底座自动学的（Claude MEMORY.md, Codex auto-instructions） 底座 home dir 中期 Alice 不接管，但通过 memory pointer 知道在哪
📚 Alice 公用 memory（薄层） 模型自决定 + 偶尔 owner 校正 ~/.alice/agents/<id>/memory/ 是 跨 task 共享的薄层，记录"agent 之间互相看得见的 learned pattern"

6.2 关键转向：Alice memory 是公用薄层，不是强干预层

设计原则修正 之前草稿把 Alice memory 写得很厚 — 每条都要 owner approve。修正：Alice memory 只是给 agent 之间共享"轻量 observation"的薄层。模型自己决定什么时候写，写完留 trace（来源 task / timestamp / 哪个 agent 写的），owner 平时不管，只有出问题时回查。

6.3 模型自决定写入 — 怎么防 poisoning

  • Memory 每条都有 metadata：来源 task / agent / timestamp / confidence
  • OTel 定期 batch（每周）跑 LLM judge 扫 memory 找矛盾 / 误导内容 → 自动 propose 给 owner review
  • UI 提供 "memory diff history" — git-style，能 revert
  • Permission mode 升级时（user_equivalent → service）invalidate 旧 memory（防 elevated agent 读到旧上下文）

8. Service Agent — admin only + 强 security 配置

Service agent 有 elevated access，比普通 agent 多一层security 必填项。

8.1 谁能创建 + 谁能担任 owner

  • 创建：只有 admin 能（不需要 superadmin）
  • Owner：必须是 admin 级别 user
  • Co-sign：另一个 admin 或 superadmin approve 才能 active

8.2 比普通 agent 多的 security 配置

必填 deny list — "明确不能做什么" 普通 agent 不需要预先列 capability，靠 intrusive filter 兜底。但 service agent 有 elevated access，必须在创建时显式列出"不能做的事"，让 admin 思考边界。这是反 confused-deputy 的核心防线。
Security 配置项普通 agentService agent
Permission modeuser_equivalent (默认)service (固定)
访问声明（能访问哪些资源）不需要必填 — 显式列出 space / tag / project
Deny list（明确不能做的事）不需要必填 — 至少 3 条
Justification（为什么需要 elevated）不需要必填 free text，写进 audit
Quarterly review不需要必填 — 季度自动创 review issue
UI 标识普通头像🛡 + auto footer "uses elevated access"
调用时 toast 提示 user不需要每次调用都提示

8.3 Deny list 例子

Service Agent: HRBot
Elevated scope: tag=hr-confidential, tag=salary, drive_space=/hr/
Deny list (必填，思考边界):
  - ❌ 不读取员工个人 chat / 私人 drive
  - ❌ 不向 workspace 之外的外部系统 export 数据
  - ❌ 不 修改任何 user.is_admin / is_superadmin
  - ❌ 不 创建新的 user / workspace
  - ❌ 不 跨 workspace 访问 (虽然内部工具一般也跨不了)
  - ❌ 不 调用任何 destructive tool (DROP / DELETE / rm -rf)
Justification:
  HR 需要一个 agent 能回答员工关于 handbook / policy 的问题，
  user 直接问没权限读 handbook 全文，agent 替它读但只输出摘要不直接 dump 原文。

8.4 季度 review 自动化

  • Cron 每季度第一周创建 review issue，assign 所有 superadmin
  • Issue 自动汇总：每个 active service agent 这季度访问次数 + scope 是否被超 + owner 是否仍在职 + deny list 是否被触发
  • 必须 close issue 才算 review 完，逾期 escalate

-- User 表加 admin 字段（admin ≠ superadmin）

-- Agent 表加 owner + permission_mode
ALTER TABLE agent ADD COLUMN owner_user_id UUID NOT NULL REFERENCES "user"(id);
ALTER TABLE agent ADD COLUMN permission_mode TEXT NOT NULL DEFAULT 'user_equivalent'
  CHECK (permission_mode IN ('user_equivalent', 'service'));
ALTER TABLE agent ADD COLUMN memory_version_hash TEXT;
-- Service agent 专用
ALTER TABLE agent ADD COLUMN service_scope_decl JSONB;       -- 访问声明
ALTER TABLE agent ADD COLUMN service_deny_list JSONB;        -- 必填 deny list
ALTER TABLE agent ADD COLUMN service_justification TEXT;
ALTER TABLE agent ADD COLUMN service_approved_by UUID REFERENCES "user"(id);
ALTER TABLE agent ADD COLUMN service_approved_at TIMESTAMPTZ;

-- Collaborator 关系
CREATE TABLE agent_collaborator (
  agent_id UUID NOT NULL REFERENCES agent(id) ON DELETE CASCADE,
  user_id  UUID NOT NULL REFERENCES "user"(id),
  added_by UUID NOT NULL REFERENCES "user"(id),   -- 当前 owner
  added_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  contributions_count INT NOT NULL DEFAULT 0,
  PRIMARY KEY (agent_id, user_id)
);

-- Memory（薄层，模型自决定写入）
CREATE TABLE agent_memory_entry (
  id            UUID PRIMARY KEY,
  agent_id      UUID NOT NULL REFERENCES agent(id) ON DELETE CASCADE,
  kind          TEXT NOT NULL,    -- 'preference' | 'observed_pattern' | 'learned_fact'
  content       TEXT NOT NULL,
  source_task_id UUID,
  source_agent_id UUID,
  written_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  confidence    NUMERIC(2,1)
);

-- Iteration request (intrusive 修改才走)
CREATE TABLE agent_iteration_request (
  id            UUID PRIMARY KEY,
  agent_id      UUID NOT NULL REFERENCES agent(id),
  proposed_by   UUID NOT NULL REFERENCES "user"(id),
  kind          TEXT NOT NULL,    -- 'prompt' | 'skill_add' | 'permission_mode' | 'owner_transfer'
  diff          JSONB NOT NULL,
  status        TEXT NOT NULL DEFAULT 'pending',
  decided_by    UUID REFERENCES "user"(id),    -- 当前 owner
  decided_at    TIMESTAMPTZ,
  decided_note  TEXT
);

-- Machine + memory pointer
ALTER TABLE machine ADD COLUMN installed_runtimes JSONB;   -- [{provider, version, installed_at}]
ALTER TABLE machine ADD COLUMN memory_pointers JSONB;       -- [{provider, path, last_seen}]
