1. Role 体系 — 两个正交维度

Role 不是 user 权限金字塔。Alice 的权限 99% 不靠 role 解决，靠 intrusive filter + audit。Role 只解决少量 system-level 状态标识。

而且 role 不是单一维度的"层级"，是两个正交维度：

🧑 维度 1: User system role

挂在 user 表，全局唯一
Role能做什么预估人数
regular 用任何普通 agent；创建自己的普通 agent ~95% 员工
admin 所有 regular 能做的 + 创建和管理 service agent（不等于 superadmin，不管 workspace / user） ~2–5 人
superadmin 全权 — workspace / user / observability 全部 admin ~1–2 人

🤖 维度 2: Per-agent relation

挂在 agent ↔ user 关系上，每个 agent 各自维护
Relation表达什么典型情况
owner 对该 agent 的行为负责任；approve intrusive action 每 agent 仅 1 人
collaborator co-work 这个 agent；记录贡献；非 intrusive 修改可以直接做 每 agent 0~N 人
这两个维度不要混在一起 admin / superadmin 是 user 的属性（user 表字段），owner / collaborator 是 user 跟 specific agent 的关系（关系表里的行）。一个 user 可能是 admin（全局），同时是 agent-A 的 owner、agent-B 的 collaborator、跟 agent-C 没关系。这四件事互不影响。

2. Owner — 责任主体，不等于 creator

核心 invariant 每个 agent 必有一个 named owner（自然人）。Owner 不必是 agent 的 creator，不必是 agent 跑的 machine 的拥有者。Owner 是当前对 agent 行为 accountable 的人。

Owner 不是组成部分。Owner 是外挂的责任主体 — 类似 GitHub repo 有 CODEOWNERS，repo 的"内容"是 code，owner 是对 code 负责的人。Repo 内容跟 owner 是两个独立东西。

2.3 Owner 责任 scope — 收窄到 intrusive action

Action需 owner approve?谁决定?
Agent 完成一个 task❌ 不需要无人 — 直接完成
Agent 写一条 conversation memory（task 内 short-term）❌ 不需要底座自己
Agent 把 learned pattern 写进 Alice 公用 memory 层❌ 不需要模型自己决定（薄层、记录来源）
Agent 调用普通 tool❌ 不需要底座自己（intrusive filter 兜底）
Agent 调用 critical tool（git push --force / DROP TABLE）✅ 需要Owner 实时 prompt

切换 model / 改 system prompt✅ 需要Owner approve iteration

Transfer ownership✅ 需要当前 owner + 新 owner accept

2.4 Owner 离职 / orphan 处理

  • HR 触发 user disable → 该 user 担任 owner 的所有 agent 进入 paused
  • Workspace admin 收到通知 + 7 天 grace period
  • Grace period 内 admin 指定新 owner 或 archive agent
  • 过期 auto-archive，audit log 永久保留
  • Service agent 不能 auto-archive（elevated access），必须 admin manual 处理

3. Collaborator — 谁 co-work 了哪个 agent

Collaborator 概念 某个 user 跟 specific agent 之间的关系 — "我帮过这个 agent 的迭代"。包括但不限于：iterate skill / 调 prompt / 提 issue / 帮 debug。Collaborator 不背 accountability，但有记录，可作为 owner transfer 候选。

3.1 Collaborator 怎么加入

  • Owner 在 agent 页面邀请：选 user → 角色 (collaborator)
  • User 提 iteration request 被 approve 后 → 自动成为 collaborator（一次性，可在 settings 取消）
  • User 显式申请加入某个 agent：Apply to collaborate button → owner approve

3.2 Collaborator 能做什么

4. Machine 维度 — 底座 install + Memory pointer

4.1 底座是可扩展集合 — 不绑死特定列表

Alice 当前已支持 11 个底座 CLI：

+ 未来任何新 CLI

底座在 daemon 端用声明式 probe pattern 注册：

// server/internal/daemon/config.go
if e, ok := probe("ALICE_FOO_PATH", "foo", "ALICE_FOO_MODEL"); ok {
    agents["foo"] = e
}

加新底座 = 加一行 probe + 在 execenv/runtime_config.go 加一个 case。不需要改 server 侧任何业务逻辑。

设计目标 未来加新底座对系统是纯加法。Memory pointer / sandbox policy / install command 这些都用同一抽象表达，新底座只需要：(a) 注册 probe，(b) 提供 install command 模板，(c) 声明 memory path，(d) 声明 sandbox 兼容性。其他系统逻辑（owner / collaborator / audit / intrusive filter）跟底座完全解耦。

4.2 Machine 详情页的 "Install Runtime" 区块设计

1选底座

  • 下拉显示 alice 当前支持的所有底座
  • 每个显示：当前版本、平台支持、官方 docs link
  • 检测 machine 当前已装版本（如有）
  • 多选 — 一次装多个底座

2一键生成 install 命令

  • Alice 根据 machine OS / arch / 底座 metadata 生成 install command
  • 员工本地终端执行（不让 daemon 自己装，安全考虑）
  • 或者生成可点击的 macOS pkg / Linux deb

3自动绑定 token

  • 装完后 alice 检测 + 引导 OAuth flow
  • Token 安全存进 keychain / Linux Secret Service
  • 不存 alice DB（user-level secret 不应集中存）

4注册 memory pointer

  • 每个底座有自己的 memory 路径声明（~/.claude/、~/.codex/、…）
  • Alice 把这些 path + version + last-sync 记录到 machine.memory_pointers
  • Alice 不复制不集中存内容 — 只是 pointer

6. Memory — Alice 是 pointer 层 / 元数据层

6.1 设计原点：能力下放底座，alice 只做最上层 pointer

关键架构判断 底座（claude / codex / hermes / …）每个都自带 memory 系统 — 写 jsonl、写 markdown、维护 conversation history 等。Alice 不要试图替代或重写这些。Alice 只做最上层一层 metadata + pointer 索引，让不同 agent 之间能互访彼此的 memory，让 task 结束后有统一地方查询 memory 状态。实际 memory 内容仍在底座目录里，alice 不集中存。

6.2 Alice memory pointer 层的职责

职责具体内容
📍 Metadata 索引每个 agent × machine × 底座的 memory 在哪里、版本、schema、last-modified — 这些是metadata，不是内容
🔗 跨 agent 访问入口agent A 想读 agent B 的 memory？通过 alice 元数据找到 B 的 memory 位置 + 调底座原生 API 读取
📊 统一查询面板所有 agent 的 memory 状态在一个 UI 里能看：哪些 agent 学了什么、memory size、conflict 检测
🔄 跨 machine 同步同一个 agent 在多个 machine 跑过 → alice 元数据知道有 N 份 memory，按需 sync（底座侧实现，alice 只编排）
🗑 生命周期管理Agent archive 时统一删除所有 machine 上的 memory（alice 通过元数据知道哪里有，发指令给底座清理）

6.3 数据模型

// alice DB 里只存 metadata，不存 memory 内容
CREATE TABLE agent_memory_pointer (
  agent_id      UUID NOT NULL,
  machine_id    UUID NOT NULL,
  provider      TEXT NOT NULL,        -- 'claude' | 'codex' | 'hermes' | ...
  memory_path   TEXT NOT NULL,        -- '~/.claude/projects/<hash>/memory/'
  schema_version TEXT,                 -- 底座 memory schema 版本
  last_modified TIMESTAMPTZ NOT NULL,
  size_bytes    BIGINT,
  tags          TEXT[],                -- 'preferences' | 'learned_pattern' | 'context'
  PRIMARY KEY (agent_id, machine_id, provider)
);

-- agent X 想读 agent Y 的 memory:
-- 1. alice 查 agent_memory_pointer where agent_id = Y
-- 2. 返回 list of (machine_id, provider, memory_path)
-- 3. agent X 通过自己的底座 API 读这些 path (跨底座可能需要 adapter)

6.4 跨底座 memory 互通问题

Claude 写的 memory 跟 codex 写的 schema 完全不同。Alice 不做 schema 转换 — 跨底座访问由调用方负责。常见处理：

  • 同底座 agent 互访：直接读对方 memory 文件（同 schema）
  • 跨底座 agent 互访：通过 LLM 概括 → 作为 plain text context 注入。Alice 提供 helper 工具但不强制
  • 不存在的 case：Alice 自己 parse 不同底座 memory 然后 unified UI 展示。不要做，schema drift 维护成本爆炸

8. Service Agent — admin only + 强 security 配置

8.1 谁能创建 + 谁能担任 owner

  • 创建：只有 admin 能（不需要 superadmin）
  • Owner：必须是 admin 级别 user
  • Co-sign：另一个 admin 或 superadmin approve 才能 active

8.2 比普通 agent 多的 security 配置

必填 deny list — "明确不能做什么" 普通 agent 不需要预先列 capability，靠 intrusive filter 兜底。但 service agent 有 elevated access，必须在创建时显式列出"不能做的事"，让 admin 思考边界。这是反 confused-deputy 的核心防线。
Security 配置项普通 agentService agent
Permission modeuser_equivalent (默认)service (固定)
访问声明（能访问哪些资源）不需要必填 — 显式列出 space / tag / project
Deny list（明确不能做的事）不需要必填 — 至少 3 条
Justification（为什么需要 elevated）不需要必填 free text，写进 audit
Quarterly review不需要必填 — 季度自动创 review issue
UI 标识普通头像🛡 + auto footer "uses elevated access"
调用时 toast 提示 user不需要每次调用都提示

8.3 Deny list 例子

Service Agent: HRBot
Elevated scope: tag=hr-confidential, tag=salary, drive_space=/hr/
Deny list (必填，思考边界):
  - ❌ 不读取员工个人 chat / 私人 drive
  - ❌ 不向 workspace 之外的外部系统 export 数据
  - ❌ 不 修改任何 user.is_admin / is_superadmin
  - ❌ 不 创建新的 user / workspace
  - ❌ 不 跨 workspace 访问 (虽然内部工具一般也跨不了)
  - ❌ 不 调用任何 destructive tool (DROP / DELETE / rm -rf)
Justification:
  HR 需要一个 agent 能回答员工关于 handbook / policy 的问题，
  user 直接问没权限读 handbook 全文，agent 替它读但只输出摘要不直接 dump 原文。

8.4 季度 review 自动化

  • Cron 每季度第一周创建 review issue，assign 所有 superadmin
  • Issue 自动汇总：每个 active service agent 这季度访问次数 + scope 是否被超 + owner 是否仍在职 + deny list 是否被触发
  • 必须 close issue 才算 review 完，逾期 escalate

9. Audit Observability — 用开源 audit 专用工具

9.1 选型 — 跟应用观测分离

关键原则 权限审计 (audit) 跟应用观测 (APM / error tracking / metrics) 必须用不同工具。Audit 是合规级需求 — append-only / 不可改 / 长期保留 / 强查询能力。Datadog / Sentry / Grafana APM 等通用观测工具不为 audit 设计。
方案类型UILicense推荐度
Graylog Open 专门 log / audit management ✅ 自带专用 audit UI、search、alert SSPL（开源足够） 首选
OpenSearch + OpenSearch Dashboards Elastic fork，AWS 维护 ✅ Kibana 同等 Apache 2.0 次选
Wazuh SIEM (security event mgmt) ✅ 有专门 audit dashboard GPL 如果未来要扩到 security event
Loki + Grafana 通用 log + 通用 BI ⚠️ Grafana 不专门为 audit 设计 AGPL 已有 Grafana stack 时可考虑
OpenTelemetry + 通用 backend 通用 telemetry SDK ❌ 通常 backend 同时被业务观测共用，audit 跟 noise 混在一起 — 不推荐 — 违反隔离原则

9.2 推荐：Graylog Open

  • 专门为 audit / log management 设计 — 不是 generic time-series
  • UI 第一公民：search query / saved dashboards / alert rules 都为 audit use case 调优
  • Stream-based 模型：audit event 流式进 Graylog，可分 stream 给不同 audience（superadmin / owner / compliance reviewer）
  • Role-based access：Graylog 自己的 RBAC 让 owner 只看自己 agent，admin 全局，compliance 看 service agent
  • Long-term retention 配置直接，符合 audit 合规需求

-- Collaborator 关系
CREATE TABLE agent_collaborator (
  agent_id UUID NOT NULL REFERENCES agent(id) ON DELETE CASCADE,
  user_id  UUID NOT NULL REFERENCES "user"(id),
  added_by UUID NOT NULL REFERENCES "user"(id),
  added_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  contributions_count INT NOT NULL DEFAULT 0,
  PRIMARY KEY (agent_id, user_id)
);

-- Memory pointer (alice 只存 metadata)
CREATE TABLE agent_memory_pointer (
  agent_id      UUID NOT NULL REFERENCES agent(id) ON DELETE CASCADE,
  machine_id    UUID NOT NULL REFERENCES machine(id),
  provider      TEXT NOT NULL,
  memory_path   TEXT NOT NULL,
  schema_version TEXT,
  last_modified TIMESTAMPTZ NOT NULL,
  size_bytes    BIGINT,
  tags          TEXT[],
  PRIMARY KEY (agent_id, machine_id, provider)
);

  kind          TEXT NOT NULL,
  diff          JSONB NOT NULL,
  status        TEXT NOT NULL DEFAULT 'pending',
  decided_by    UUID REFERENCES "user"(id),
  decided_at    TIMESTAMPTZ,
  decided_note  TEXT
);
