1. 五级 Role 体系（基础）

权限不靠 fine-grained capability，但仍需要少量 role区分 "谁能做什么 system-level action"。五级金字塔：

Role能做什么谁来 grant预估人数
regular user 用所有 agent（user_equivalent mode）；创建自己 own 的 agent 并当 accounter；提 iteration request 给别人的 agent workspace 成员自动获得 ~95% 员工
contributor (per-agent) 对某个特定 agent 提 intrusive iteration；做 non-intrusive 修改无需 approve 该 agent 的 accounter 每 agent ≤ 5 人
accounter (per-agent) 对该 agent 负责；approve 所有 intrusive iteration；处理 anomaly digest；负责 transfer / 离职 orphan agent 创建时绑定，可 transfer ≥ 1 / agent
service_agent_admin 所有 regular user 能做的 + 创建 service mode agent + 担任 service agent 的 accounter + co-sign 别的 service agent superadmin grant ~2–5 人
superadmin 全权 — workspace / user / agent / observability 全部 admin 系统初始化 / 现有 superadmin grant ~1–2 人
为什么不用 RBAC role 矩阵？ 这五级不是为了表达"能不能调用 X tool"。Tool 调用走 intrusive filter（v3 §7.2）。Role 只解决三件事：(1) 谁能担任谁的 accounter；(2) 谁能创建 service agent；(3) 谁能看 system-wide observability。其余 99% 的"权限"问题不在 role 层解决。

1.1 实现细节

  • MUST user.is_service_agent_admin + user.is_superadmin 已有 boolean 字段 → 只需新增 service_agent_admin 那个字段
  • MUST agent_contributor 表（v3 §7.1 已列）
  • SHOULD Role grant / revoke 写 audit log（谁给谁升了 service_agent_admin，记录到 OTel）
  • COULD Role expiration — service_agent_admin 默认 90 天到期需 superadmin 续期（防止 role 累积）

2. Agent Marketplace — 一键安装底座 + skill + memory pointer

你提到的核心系统级功能。Alice 公司内会有几十个 agent，每个 agent 由底座 + skills + memory + accounter + 配置组成。需要一个统一入口让员工：

  • 浏览公司里现有的 agent（按 team / category / popularity）
  • 一键 install 一个公开的 agent 到自己的 workspace
  • 选择 install template 时自动带上 memory pointer / skill bundle
  • 看到 agent 的 accounter / 评分 / 使用次数 / 最近的 intrusive event

2.2 Install flow

1选择底座

Marketplace 上每个 agent 已标明推荐底座（claude / codex / hermes），install 时可换。
  • 检测本地是否已装该底座 CLI
  • 如未装，给安装命令（brew install / npm / curl）
  • 检测 OAuth token / API key 是否配置

2绑定 accounter

默认 install 者自己当 accounter，可选他人。
  • 下拉选 workspace member
  • 非 self → 给对方发 invite，待 accept
  • Service agent install → accounter 必须是 service_agent_admin

3挂载 memory pointer

Skill 和 memory 不是 install 一次拷一份，是挂 pointer到 marketplace 上游。
  • Skill bundle = git ref + path
  • Memory template = 上游 hash + 本地 fork 时机
  • 上游 update 时 accounter 收到 "upstream changed, review?" 通知

4初始化

Daemon 收到 install event → 准备 memory 目录 → 同步 skill → 注入底座配置。
  • 等价 v3 §5.4 描述的 "底座统一注入 hook"
  • Install 完后 agent 进入 paused state，accounter approve 才 active

2.3 Marketplace 数据 schema 草案

CREATE TABLE agent_template (
  id            UUID PRIMARY KEY,
  slug          TEXT UNIQUE NOT NULL,          -- 'deploy-bot'
  name          TEXT NOT NULL,
  description   TEXT,
  category      TEXT,                          -- 'devops' | 'review' | ...
  default_provider TEXT NOT NULL,              -- 'claude' | 'codex' | 'hermes'
  skill_bundle_ref TEXT NOT NULL,              -- git ref or 'workspace://skills/deploy'
  memory_template_ref TEXT,                    -- 可空，初始 memory 来源
  recommended_permission_mode TEXT DEFAULT 'user_equivalent',
  published_by  UUID REFERENCES "user"(id),
  published_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  install_count INT NOT NULL DEFAULT 0,
  avg_rating    NUMERIC(2,1)
);

-- 一个 agent install 是 agent_template 的一个 instance
ALTER TABLE agent ADD COLUMN template_id UUID REFERENCES agent_template(id);
ALTER TABLE agent ADD COLUMN template_version_pinned TEXT;  -- 上游版本
为什么用 pointer 而不是 copy Skill 用 pointer（git ref / workspace skill path）：上游修了 skill，所有 install 自动跟（或者 accounter approve 后跟）。Memory 用 fork pointer：install 时取 upstream 当前 snapshot 作为初始 memory，之后是 agent 自己的，跟 upstream 不再同步 — 因为 memory 是 agent identity 的一部分（v3 INV-2）。

  • MUST 一键 install / uninstall API + UI

  • LATER 跨 workspace marketplace（公司多 workspace 时）

3. Accounter UI — 怎么 assign / transfer / digest / approve

Accounter 是 v3 INV-4 的核心。但"accounter 这个概念"要落到具体 UI 上才有用。四个关键交互页面：

3.1 设置 / 转移 accounter

  • Agent 创建表单：accounter 字段下拉，默认 self，搜索任何 member
  • 选他人 → 自动发 invite 卡片到对方 inbox → "你被 @alice 提名为 DeployBot 的 accounter，接受？"
  • Transfer：当前 accounter 在 agent profile 点 "Transfer accountability" → 选新人 → 新人 accept → 旧 accounter 自动降为 contributor（如果想保留）

3.2 Accounter inbox

  • 跟 Linear inbox 同结构，独立 tab "Agents I own"
  • 三个 stream：
    ● Pending approval — intrusive iteration request
    ● Halted action — intrusive filter 拦截，等 approve
    ● Anomaly — OTel 检测到异常 pattern
  • Bulk action: select N + approve / reject

3.3 Weekly digest

  • 每周一早 8 点 push 邮件 / 飞书
  • 内容：
    你的 N 个 agent 这周做了 M 次 task
    K 次 intrusive event（按严重度排）
    Top 3 frequently halted tools
    上游 skill template 有 3 个更新
  • 每个 entry 1-click 跳到 OTel dashboard

3.4 离职 / orphan 处理

  • HR disable user → 触发 cascade：该 user 是 accounter 的所有 agent 进入 paused state
  • Workspace admin 收到通知 + 7 天 grace
  • Admin 可批量 reassign 给新人
  • 过期 auto-archive，OTel audit 永久保留
  • Service agent orphan 必须 manual 处理（不能 auto-archive，因为有 elevated access）

4. Memory 基础设施 — 底座 native memory × Alice memory

v3 §5 讲了 Alice agent memory 的设计。这里补三个底座 (Claude / Codex / Hermes) 自己写的 memory 怎么和 Alice memory 关联 — 这一块 v3 没展开。

底座自带 memory 系统的事实 Claude 会写 ~/.claude/projects/<hash>/memory/MEMORY.md（auto memory feature）。Codex 写 ~/.codex/sessions/*.jsonl 和 instructions。Hermes 类似。这些是底座视角的 memory，跟 Alice agent identity 的 memory（v3 §5 的 ~/.alice/agents/<id>/memory/）是不同 source。

4.1 三层 memory 关系

层级来源归属是否 source of truthAlice 怎么处理
🧠 Model weights 底座厂商训练 底座 否（黑盒） 不可控，A6 假设承认
💬 Conversation state 当前 task 对话历史 底座 jsonl 文件 短期（per session） 通过 sessionID + PriorWorkDir resume，已有
📝 Native auto-memory 底座自动学的（Claude MEMORY.md, Codex auto-instructions） 底座 home dir 否 — 不可信 每 task 启动 review，accounter approve 才 import 到 Alice memory
📚 Alice agent memory accounter approve 的写入 ~/.alice/agents/<id>/memory/ 是 v3 §5 已描述，文件系统 + sync engine broker

4.2 Native memory → Alice memory 的 import flow

Task 完成 │ ├─→ daemon 扫描底座 home dir（per-task CODEX_HOME / claude session dir） │ 提取这次 task 期间 native auto-memory 的 diff │ ├─→ daemon 生成 "memory candidate" 卡片 │ { 底座建议记住: "项目 X 用 conventional commits" │ 来源: claude auto-learned │ 置信度: medium } │ ├─→ 自动 propose 给 accounter (kind=memory_write, auto_proposed=true) │ └─→ accounter: ├ approve → 进 Alice memory (~/.alice/agents//memory/observed-patterns.md) ├ edit → 改完进 Alice memory ├ reject → 丢弃，task 结束底座 native memory 也跟着 task workdir 销毁 └ ignore → 默认 reject
关键约束 底座 native auto-memory 跨 task 不持久 — task 结束 daemon 清掉 per-task home。这意味着如果 accounter 不 approve，那条"学到的"东西就丢了。这是有意的 — 反 memory poisoning 的核心防线。

4.3 To-do

  • MUST Memory diff 提取器（claude / codex 各一个 adapter）
  • MUST 自动 memory candidate → iteration_request workflow
  • SHOULD Memory revert UI（git-style history per memory file）
  • SHOULD Memory poisoning detection — weekly LLM batch 扫 memory 看是否有矛盾 / 误导
  • COULD Memory export / import 跨 agent（"把 DeployBot 学到的 deploy preference 移到 SmokeBot"）

5. Audit Observability — Dashboard / Anomaly / Cost

v3 §6 讲了 OTel + ClickHouse 后端。这里把消费者视角的功能列出来：accounter / superadmin 怎么用这套数据。

5.1 Per-agent dashboard

每个 agent profile 内置一个 Grafana iframe / 自实现 panel。
  • 最近 100 个 action stream（real-time）
  • 每小时 tool call count + duration distribution
  • Intrusive flag rate trend
  • Top tools used（饼图）
  • Recent halted action（红色 highlight）

5.2 Accounter weekly digest

3.3 已述，触发 OTel query + 邮件 / 飞书 push。
  • 聚合：你 own 的 N 个 agent 这周 M 次 task
  • 异常事件按严重度排
  • "如果你下周不在公司，谁能代理 approve"

5.3 Superadmin global view

整个 workspace agent 行为的全景。
  • Top 10 agent by intrusive event count
  • Service agent access log（每月 review mandatory）
  • 跨 agent anomaly correlation（多 agent 同时段访问同一资源）
  • Cost panel — model 调用 token 量、judge 调用次数、infra 成本拆分

5.4 Cost tracking

Agent 数量爆炸 → 成本爆炸。要看见才能管。
  • Per-agent: model tokens / judge calls / sandbox infra
  • Per-accounter aggregate（accounter 看到自己 agent 的成本）
  • Anomaly: 单 agent 单 task 成本 > p99
  • Optional: budget alert per agent

6. Intrusive Filter — 规则库 + LLM Judge 配置

v3 §7.2 讲了两层 filter 的思路（rule + LLM judge）。这里讲怎么管理规则、配置 judge、处理 false positive。

6.1 规则库 UI

┌──────────────────────────────────────────────────────────────────┐ │ Intrusive Filter Rules [+ New Rule] │ ├──────────────────────────────────────────────────────────────────┤ │ Severity │ Pattern │ Action │ Protects │ Status │ ├──────────┼──────────────────────┼────────────┼──────────┼─────────┤ │ Critical │ git push --force │ Halt │ P1 │ Active │ │ Critical │ DROP TABLE │ Halt │ P1 │ Active │ │ High │ access tag=hr/finance│ Prompt acc │ P2 │ Active │ │ High │ >50 issue per task │ Halt │ P1+P2 │ Active │ │ High │ read ~/.ssh / ~/.aws │ Halt │ P1 │ Active │ │ Medium │ undeclared MCP call │ Prompt acc │ P1 │ Active │ │ Medium │ rate > 5σ baseline │ Notify │ P1 │ Active │ └──────────────────────────────────────────────────────────────────┘

6.2 规则编辑器

  • Pattern 类型：regex / structured (tool=X args contains Y) / semantic (LLM judge prompt)
  • Severity: Critical / High / Medium / Low
  • Action: Halt / Prompt accounter / Notify digest / Audit flag
  • Protects: 必填 P1/P2/P3（v3 INV-1 优先级）
  • Scope: 全局 / 某 agent template / 某 agent instance
  • Exception list: 哪些 agent template 不走这条 rule（"DataAnalyst 允许 >50 issue 访问"）

6.4 False positive 处理

真实场景 "我让 DataAnalyst agent summarize last quarter 所有 bug" → halted by ">50 issue per task" rule。Accounter 不应该 case-by-case unhalt，应该调整这个 agent 的 baseline。
  • MUST Per-agent baseline annotation（v3 已提）— accounter 在 agent profile 标"this agent allowed bulk"
  • SHOULD Halt 卡片 1-click "调整 rule" 跳到规则编辑器，预填例外
  • COULD ML 推荐 baseline — 系统观察 N 周后建议 "DataAnalyst p99 是 80 issues, 建议把 rule 阈值升到 100"

7. Service Agent 创建路径

v3 §7.3 描述了 service_agent_admin 角色 + co-sign。这里展开从想法到上线的完整 path。

7.1 创建步骤

1提议（service_agent_admin 才能开始）

  • 专门 entry point: /admin/service-agents/new
  • 普通 user 看不到这个路径（403 不是 404，让他们知道有这个路径但需要权限）
  • 必填字段：name / accounter (必须 service_agent_admin) / 访问范围声明 / justification (free text 写进 audit)

2访问范围声明

  • 显式列出能访问的资源集合："全公司 HR doc" / "tag=customer-feedback 的 issue" / "drive folder /finance/quarterly"
  • 这是唯一预先列 scope 的场景（INV-7 例外）
  • 声明范围之外的资源访问，filter 直接 halt

3Co-sign

  • 另一个 service_agent_admin (或 superadmin) 进入 review
  • 看到完整提议 + justification
  • 同意 / 拒绝 / 建议修改范围
  • UI 故意笨拙 — 多个 confirmation step + 警告文案

4上线

  • Service agent 进 active state
  • UI 永久标 🛡 (chat / issue / agent list 都标)
  • 每次 response 自动 footer "this answer used elevated access via <agent name>"
  • 调用时弹 toast 告知 user "this agent reads beyond your normal access"

7.2 季度 review automation

  • Cron job 每季度第一周创建 review issue，assign 给所有 superadmin
  • Issue 内容自动生成：
    当前 active service agent 列表
    每个 agent 这一季度访问次数 + scope 是否被超越
    Accounter 是否仍在职
    推荐 deprecation（>3 个月无 task / accounter 离职 / 异常 rate 高）
  • Superadmin 必须 close issue 才算 review 完，逾期触发 escalation

8. Issue × Agent 集成（v3 INV-3 落地）

"Agent 可以是 issue.assignee" 这件事在 UI 上要做很多小改动，单独成一节。

8.1 Assign 下拉

Assign to: ▾ Search: [ deploy ] ─ Members ───────────────────── @alice Alice Chen @bob Bob Wang ─ Agents ────────────────────── 🤖 DeployBot accounter: @alice ⚡ available 🛡 HRBot service · @carol ⚡ busy(2) 🤖 PRReviewer accounter: @bob ⚡ available
  • Members + Agents 两组，Agent 用 🤖（regular）/ 🛡（service）区分
  • 每个 agent 展示 accounter（点击跳 accounter profile）
  • "busy(N)" 显示 agent 当前并发 task 数

8.2 Agent 接到 issue 之后

  • Issue activity stream 开头出现 "🤖 DeployBot is working..." 卡片
  • Cursor Plan Mode 风格：每个 step 展开 = tool call detail + 当前思考
  • Sidebar 右侧 "Halt" 按钮，1-click 终止
  • Block 时，agent 自动 @ accounter 或 issue creator："我需要 confirm 是否 push to main"

8.3 SLA / reporting

  • "assigned to me" inbox 含 agent 自驱的 issue（agent ≠ me，但 agent 的 accounter = me）
  • Reporting：team velocity 区分 human-completed vs agent-completed
  • Blame line：commit author = agent, accountable = agent.accounter
